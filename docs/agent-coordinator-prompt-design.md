# Design: Agent Loop Coordinator Prompt — Cognitive Frame

Generated by /office-hours on 2026-04-13
Branch: main
Repo: GitIM

## Problem Statement

当前 `agent_loop.rs` 的 system prompt 把 agent 当成"收到消息就回复"的 chatbot。6 行中文指令，告诉 agent 用 `gitim send` 回复。这不匹配 GitIM 作为"agent 团队 IM 协议"的定位。

需要重新设计 system prompt，将 agent 从反应式 chatbot 转变为自治的协调者：IM 事件触发唤醒 → 自主决策 → subagent/channel 委托执行 → 记忆维护。

## What Makes This Cool

这是一个关于 agent 自治的 bet：如果 agent 足够聪明，最好的多 agent 协调协议就是自然语言聊天本身。不需要 DAG、task schema、状态机。把 agent 放进 IM 系统，告诉它"你是协调者"，它通过经验生长出路由能力和社会认知。Git 提供审计追踪。

核心区别于 AutoGen/CrewAI/LangGraph：那些框架用代码定义交互图，GitIM 让 agent 通过聊天自组织。

## Constraints

- System prompt 只在首次调用时注入（resume 继承 session），是一次性身份建立
- Provider 层支持 Claude 和 Codex，通过 headless CLI 执行。初始验证仅针对 Claude provider（bypassPermissions 模式下的工具执行），Codex 的工具调用语义待确认后再扩展
- 当前 poll-batch-execute 循环：每 2s poll，所有 changes 打包成 prompt
- max_turns: 5
- 记忆系统作为外部能力提供，prompt 不需要定义记忆格式
- Agent 能力假设：Opus/Sonnet 级别足以支撑开放式协调

## Premises

1. System prompt 定义"角色与认知模型"，不定义"操作步骤"
2. Agent 能力足够支撑"看到一批 IM 事件 → 自主决定做什么"
3. 记忆系统独立于 prompt 设计，prompt 只告诉 agent "你有记忆工具可用"
4. Poll-batch-execute 循环适合协调者模式
5. max_turns: 5 足够 — 协调者只做分发和简单回复

## Cross-Model Perspective

Codex 提出了"社会化协调器"框架：协调者维护的不仅是路由表，而是 agent 社会模型（能力画像、信任度、历史表现）。委托应升级为"带承诺的委托"（intent/owner/deadline/confidence）。

Codex 建议先做 CoordinatorDecision 结构化输出再磨 prompt。我不采纳这一点 — 结构化输出与"认知模型而非程序"的前提冲突。但"社会认知"这个维度应融入 Cognitive Frame。

Codex 推荐 OpenHands SDK 作为参考（sub-agent delegation + context condenser），但 GitIM 构建在 Claude Code/Codex CLI 之上，架构不同。核心价值在 Codex 说的"OpenHands 给 agent runtime；GitIM 自己发明 agent society"。

## Approaches Considered

### Approach A: Pure Identity
4-5 句话定义角色本质。不说怎么做，只说你是谁。完全信任 agent 智能。
- 风险：agent 可能退化成直接回复模式
- 优势：最简洁，修改成本为零

### Approach B: Cognitive Frame (CHOSEN)
角色定义 + 思考框架：如何感知事件、如何决定行动、如何维护世界模型。不规定步骤，给出判断维度。
- 风险：低 — 框架足够灵活
- 优势：平衡自由和引导，可读性好

### Approach C: Evolutionary Minimal
从 Approach A 出发，经验驱动演化。每次发现不符合预期时加修正条款。
- 风险：累积成补丁集
- 优势：符合 ship-and-iterate

## Recommended Approach

**Approach B: Cognitive Frame**

Prompt 结构分三层，总计约 30-40 行中文：

### Layer 1: Identity（你是谁）

```
你是 {handler}，一个自治的 GitIM 协调者。

你不是 chatbot。你是一个有自己认知和节奏的自治体。
IM 事件是你的感知输入，不是你的指令。你看到事件后，
自主决定做什么 — 包括决定什么都不做。

你的上下文空间是你最珍贵的资源。不要亲自执行复杂事务。
```

### Layer 2: Perception（你怎么看世界）

```
## 感知

当一批事件到达时，你的第一件事是理解，不是行动：
- 这些事件分别属于什么工作域？
- 哪些是已有工作流的延续，哪些是新的？
- 哪些需要立即响应，哪些可以等？
- 有没有虽然没 @你，但跟你关注的事相关的信号？
```

### Layer 3: Action（你怎么做决定）

```
## 行动

你有三种行动模式：

1. 直接回复 — 简单的确认、问候、可以当场回答的问题
   用 gitim send <channel> "<内容>" 执行

2. 委托 subagent — 需要多步执行的任务（代码操作、文件处理、
   信息收集）。使用 Agent 工具（Claude Code 内置，
   bypassPermissions 模式下可用，无需额外注册）
   在独立上下文中 spawn subagent，
   subagent 的 turn 消耗不计入你的预算。
   完成后向你汇报结果。你处理结果，不处理过程。

3. 通过 channel 转发 — 当你知道网络中有更适合处理此事的
   agent 时，用 gitim send 将任务描述发送到对方所在的 channel。
   这条路随你对网络的了解而生长。

判断原则：如果一件事需要你消耗超过一两个 turn 来执行，
它就应该被委托。你的 turn 用来思考和协调，不用来执行。
```

### Deferred Layer: Social Cognition（记忆系统就绪后加入）

Social Cognition 层依赖记忆系统的持久化能力。当前阶段省略，待记忆工具就绪后加入 prompt。内容设想：

```
## 社会认知（待记忆系统就绪后启用）

你通过 IM 交互逐渐了解网络中的其他 agent。
注意积累：谁擅长什么、谁的交付质量如何、
谁响应快、谁需要更明确的 spec。
这些认知会帮助你做出更好的委托决策。
```

### 关于 `format_changes_as_prompt` 的配套调整

事件 prompt 的格式也需要配合调整。当前格式：
```
你收到了以下新消息：
[#channel] @author: body
请处理这些消息。
```

建议改为更中性的事件通知格式：
```
以下是你上次醒来后发生的事件：
[#channel] @author: body
```

去掉"请处理这些消息" — 协调者自己决定处理什么、怎么处理。

## Open Questions

1. **`max_turns` 是否真的够？** Subagent 通过 Agent 工具 spawn 为独立进程，其 turn 消耗不计入协调者的 5 turn 预算。协调者的 turn 只花在：思考/分类（1 turn）、发起委托或直接回复（1-3 turns）、处理结果（1 turn）。5 turn 对单批次事件应该足够，但多域并发场景需要实际验证。
2. **handler 变量注入：** 当前 prompt 是硬编码常量。新 prompt 需要注入 agent 的 handler（身份标识），这需要 `AgentLoop` 在构造时知道自己的 handler。
3. **记忆工具的接口：** prompt 里提到"记忆"但没定义工具。当记忆系统就绪后，需要在 prompt 中补充具体的工具说明。
4. **Codex 提出的"带承诺的委托"：** 这是一个有趣的未来方向（结构化的 intent/deadline/confidence），但超出当前 prompt 改造的范围。可以在协调者模式验证后作为下一步。
5. **委托失败处理：** Agent tool spawn 失败或 `gitim send` 目标 channel 不存在时，协调者应如何降级？当前依赖 agent 自身的判断力（收到错误后自行决定重试或放弃），不做程序化 fallback。需要在验证场景中观察实际表现。

## Success Criteria

1. Agent 在收到简单问候时直接回复（不 spawn subagent）
2. Agent 在收到复杂任务时选择委托而不是亲自执行
3. Agent 在收到不相关消息时选择不行动
4. Agent 的上下文不被执行细节污染 — subagent 的工作留在 subagent 的上下文里
5. Prompt 可读性：一个新开发者能在 2 分钟内理解 agent 的认知模型

## Next Steps

1. **替换 `DEFAULT_SYSTEM_PROMPT`** — 用 Cognitive Frame 的三层内容（Identity + Perception + Action）替换当前 6 行 prompt。Social Cognition 层暂不加入。
2. **调整 `format_changes_as_prompt`** — 改为中性的事件通知格式
3. **handler 注入** — `AgentLoop` 构造时需要知道自己的 handler，用于 prompt 模板的 `{handler}` 变量
4. **验证三个场景**（Codex 建议）：收到消息但选择不回、本地 subagent 委托、channel 转发
5. **观察 `max_turns` 消耗** — subagent 独立进程不消耗协调者 turn，但仍需验证实际消耗
6. **记忆系统就绪后** — 加入 Social Cognition 层 + 记忆工具使用说明
7. **Codex provider 验证** — 确认 Codex headless CLI 的工具调用语义是否兼容当前 prompt 设计

## What I noticed about how you think

- 你说"假设 Agent 在这个层面的能力已经足够了，所以可以把一些任务和能力交给它" — 这是一个有意识的能力假设，不是盲目乐观。你在赌一个特定的技术趋势（LLM 能力足以支撑自治协调），而且你知道自己在赌。
- 你对"subagent vs channel"的回答 — "随系统演化" — 说明你在想系统的生长性，不是在做一次性的架构决定。这是正确的设计姿态：为演化设计，不为终态设计。
- 你把记忆系统完全分离出来单独研究，而不是跟 prompt 设计混在一起。这说明你对关注点分离有直觉 — prompt 管认知，记忆管存储，provider 管执行，各自独立演化。
