> 整合人：ux-architect | 日期：2026-03-17
>
> 基于 5 个预研方案（realtime-bridge、web-frontend、TUI、desktop、UX 设计）的交叉 review 结果，整合为 3 套完整可行方案。

## 一、各方案 Review 摘要

### 1. realtime-bridge（WebSocket Bridge）
- **评分**: 3.60/5.00
- **核心价值**: 零侵入 daemon、合规性完整、mock 模式支持前端独立开发
- **关键问题**: 初始消息全量推送冲击、channelCursors 全局共享（多客户端丢消息风险）、缺少通知分级元数据
- **改进建议**: subscribe 支持 since 参数、推送区分 history/live、增加 mentions 和 is_human 字段

### 2. web-frontend（Svelte 5 + Vite）
- **核心价值**: 编译时框架零运行时开销、完全掌控 UI 渲染逻辑、分栏线程视图
- **关键问题**: 虚拟滚动缺失（AI 消息量巨大时 DOM 会崩溃）、Agent 颜色分配策略未实现、Markdown 渲染未集成
- **亮点**: 分栏线程可视化（主列 + 右侧线程面板）与 UX 设计的 Thread Reader 模式高度吻合

### 3. TUI（Ink / React for CLI）
- **评分**: 2.90/5.00
- **核心价值**: TypeScript 生态复用（client.ts）、子命令共存、React 组件模型开发快
- **关键问题**: 信息过载无应对（缺少 AI 消息折叠/过滤/通知分级）、人类消息未视觉突出、代码块无语法高亮
- **底线未达标**: 人类消息突出 ✗、语法高亮 ✗

### 4. desktop（Tauri 2.0）
- **核心价值**: Rust 生态协同（直接引用 gitim-core）、Unix Socket 直连、notify 文件监听实时性高、包体积 3-8MB
- **关键问题**: WebView 跨平台渲染差异（Linux WebKitGTK）、Rust 编译时间长、需维护 Rust 后端 + Web 前端两套代码
- **亮点**: 多仓库支持设计、离线体验、daemon 生命周期管理

### 5. UX 设计
- **核心价值**: 双视图架构（Mission Control + Thread Reader）、P0-P4 通知分级、Agent 状态系统、8 维评估框架
- **关键洞察**: AI 消息量是人类的 10-100 倍，信息过载是核心 UX 挑战；没有单一视图能满足全部场景

## 二、底线标准对照

所有方案必须满足的 5 条硬性要求：

| 底线标准 | Web (Svelte) | TUI (Ink) | Desktop (Tauri) |
|----------|:---:|:---:|:---:|
| point_to 关系不丢失 | ✓ 分栏展示 | 部分（仅 3 层 `>` 前缀） | ✓ 可复用 Web 层 |
| 人类消息视觉突出 | 部分（Agent 色带有设计，未区分人/AI） | ✗ 未区分 | 取决于 Web 前端实现 |
| 代码块语法高亮 | ✗ 未集成 | ✗ 仅绿色缩进 | 取决于 Web 前端实现 |
| Git 同步状态可见 | ✗ 未实现 | 部分（连接状态） | ✓ daemon 状态管理 |
| 响应式设计 (1280px+) | ✓ | ✓ 终端自适应 | ✓ |

**结论**: 没有任何单一方案完全达标。需要组合方案互补短板。

## 三、整合方案

### 方案 A：Web-First（推荐 - MVP 最快路径）

> 适合场景：快速验证、团队以 Web 开发者为主、预研阶段

#### 技术栈
```
前端框架:   Svelte 5 + Vite + TypeScript
实时通信:   realtime-bridge (WebSocket Bridge)
部署形态:   本地 Web 应用 (localhost)
运行依赖:   Node.js + gitim-daemon
```

#### 架构
```
┌─────────────────────────────────────────────┐
│          Svelte 5 Web App                   │
│  ┌─────────────┐  ┌──────────────────────┐  │
│  │ Mission     │  │ Thread Reader        │  │
│  │ Control     │  │ (分栏线程视图)        │  │
│  │ (多频道概览) │  │                      │  │
│  └─────────────┘  └──────────────────────┘  │
│            │ WebSocket                       │
├────────────┼────────────────────────────────┤
│     realtime-bridge (Node.js)               │
│     500ms 轮询 → WebSocket 推送             │
│            │ Unix Socket                     │
├────────────┼────────────────────────────────┤
│     gitim-daemon (Rust)                     │
│     .thread 文件 + Git                      │
└─────────────────────────────────────────────┘
```

#### UX 模式
- **默认视图**: Mission Control — 多频道卡片概览，快速扫描活动
- **深入视图**: Thread Reader — 点击频道/话题后切换到分栏模式（主列消息流 + 右侧线程面板）
- **一键切换**: 顶部 Tab 或快捷键在两种视图间切换
- **信息降噪**:
  - P0-P4 通知分级（@我 红色、人类消息 蓝色、参与话题 灰色）
  - 连续 5+ 条 AI-to-AI 消息自动折叠为摘要
  - 人类消息左侧彩色边框 + HUMAN 标签突出显示
- **代码块增强**: marked + highlight.js 语法高亮、行号、复制按钮
- **Agent 头像**: 基于 handler hash 生成唯一颜色 + 首字母，AI 方形 / 人类圆形

#### 需补齐的关键组件
1. **虚拟滚动** — @tanstack/virtual 或 svelte-virtual-list，处理千级消息
2. **Markdown 渲染** — marked + highlight.js
3. **通知分级逻辑** — bridge 推送增加 mentions/is_human 元数据
4. **bridge 改进** — subscribe 支持 since 参数、per-client cursor、history/live 区分

#### 优缺点

| 优势 | 劣势 |
|------|------|
| 开发速度最快，Svelte 原型已就绪 | 依赖浏览器，非原生体验 |
| bridge + Svelte 均为 TS，代码可共享 | 两个额外进程（bridge + dev server） |
| UX 设计自由度最高，可完整实现双视图 | 500ms 轮询延迟（AI 场景可接受） |
| 生态成本低，不需要 Rust 前端经验 | 无系统通知/托盘图标 |

### 方案 B：Desktop-Native（推荐 - 最佳长期方案）

> 适合场景：长期产品化、需要系统级集成、重度用户

#### 技术栈
```
前端框架:   React + Vite + TypeScript (Tauri WebView)
后端通信:   Tauri Rust 后端直连 Unix Socket + notify 文件监听
部署形态:   原生桌面应用 (.app / .deb / .exe)
运行依赖:   gitim-daemon (Tauri 可自动管理)
```

#### 架构
```
┌─────────────────────────────────────────────┐
│          Tauri 2.0 Desktop App              │
│  ┌───────────────────────────────────────┐  │
│  │  React WebView (前端 UI)              │  │
│  │  Mission Control + Thread Reader      │  │
│  │  系统通知 / 全局快捷键 / 托盘         │  │
│  └───────────────┬───────────────────────┘  │
│                  │ Tauri IPC (invoke)         │
│  ┌───────────────┴───────────────────────┐  │
│  │  Rust 后端                            │  │
│  │  ├─ daemon.rs  (Unix Socket 直连)     │  │
│  │  ├─ watcher.rs (notify 文件监听)      │  │
│  │  ├─ gitim-core (消息解析复用)         │  │
│  │  └─ commands.rs (Tauri 命令)          │  │
│  └───────────────┬───────────────────────┘  │
│                  │ Unix Socket                │
├──────────────────┼──────────────────────────┤
│          gitim-daemon (Rust)                │
│          .thread 文件 + Git                 │
└─────────────────────────────────────────────┘
```

#### UX 模式
- 与方案 A 相同的双视图架构（Mission Control + Thread Reader）
- **额外能力**:
  - 系统原生通知（P0 消息弹窗提醒）
  - 系统托盘常驻 + 未读徽标
  - 全局快捷键（如 Cmd+Shift+G 打开/切换 GitIM）
  - 多仓库切换（类似 Slack workspace）
  - 离线浏览历史消息
  - daemon 自动启动/管理

#### 实时通信优势
- **不需要 bridge**: Tauri Rust 后端直连 daemon socket，省去中间层
- **文件监听**: notify crate 提供毫秒级文件变化事件，比 500ms 轮询更实时
- **双通道**: daemon API 负责读写 + 文件监听负责变化通知，互补

#### 需补齐的关键组件
1. **前端 UI** — 可复用方案 A 的 Svelte 组件（Tauri 支持任意 Web 框架），或用 React 重写
2. **Rust 后端模块** — daemon 连接管理、文件监听、消息缓存
3. **系统集成** — 通知、托盘、全局快捷键、自动更新

#### 优缺点

| 优势 | 劣势 |
|------|------|
| 最佳用户体验（原生系统集成） | 开发成本最高（Rust + Web 双栈） |
| 实时性最好（notify 文件监听） | Rust 编译时间长 |
| 包体积小（3-8MB vs Electron 150MB+） | Linux WebKitGTK 渲染差异风险 |
| 直接复用 gitim-core，类型安全 | 需要 Rust 前端经验 |
| 离线支持、多仓库管理 | 交叉编译/CI 配置复杂 |

## 五、推荐策略：渐进式路径

**不是三选一，而是按阶段演进。**

### 阶段 1（当前 → 2 周）：方案 A — Web-First MVP

**目标**: 快速验证 UX 设计假设，尤其是双视图架构和信息降噪策略。

行动项：
1. 在 Svelte 原型基础上实现 Mission Control + Thread Reader 双视图
2. 改进 realtime-bridge：subscribe 支持 since、推送区分 history/live
3. 集成虚拟滚动（@tanstack/virtual 的 Svelte 适配）
4. 实现 AI 消息折叠 + 人类消息高亮
5. 集成 Markdown 渲染 + 代码语法高亮

**验证指标**:
- Tech Lead 能在 30 秒内了解所有频道状态（Mission Control）
- 能从任意消息追溯完整线程链（Thread Reader）
- 人类消息在 AI 消息流中一眼可见

### 阶段 2（2-4 周）：方案 C — TUI 增强（并行）

**目标**: 为键盘流开发者和 SSH 场景提供轻量替代。

行动项：
1. TUI 补齐底线标准（人类消息突出、代码高亮）
2. 增加 AI 消息折叠和消息过滤
3. 接入真实 daemon（替换 mock）
4. 与 bridge 对接（可选，提供实时推送）

### 阶段 3（4-8 周）：方案 B — Desktop-Native

**目标**: 基于阶段 1 验证过的 UX 设计，构建产品级桌面应用。

行动项：
1. Tauri 2.0 脚手架，Rust 后端实现 daemon 连接 + 文件监听
2. 将阶段 1 的 Web 前端迁移到 Tauri WebView（React 或 Svelte 均可）
3. 系统集成（通知、托盘、全局快捷键）
4. 多仓库支持
5. 自动更新和 daemon 生命周期管理

### 最终形态

```
┌─────────────────────────────────────────────────────────┐
│                    GitIM 前端矩阵                        │
│                                                         │
│  ┌─── Agent/CI ───┐  ┌── 开发者 ──┐  ┌── 所有人 ──┐   │
│  │   gitim CLI    │  │ gitim tui  │  │  Desktop   │   │

│  └───────┬────────┘  └─────┬──────┘  └─────┬──────┘   │
```

三种客户端共存，服务不同用户画像和使用场景。共享 daemon 通信层和 UX 设计语言。

## 六、关键决策记录

1. **实时层选择 WebSocket Bridge 而非 SSE**: v1 阶段零侵入 daemon 的价值大于 50ms 延迟优势
2. **Svelte 而非 React**: GitIM 的 point_to 线程模型需要完全自定义 UI，轻量框架更合适
3. **双视图而非单视图**: 信息过载是 AI IM 的核心挑战，Mission Control 解决"概览"，Thread Reader 解决"深入"
4. **TUI 保留 Ink 原型但规划迁移**: Ink 快速验证交互设计，ratatui 是长期高性能方案
5. **Desktop 选 Tauri 而非 Electron**: Rust 生态协同 + 包体积优势决定性
6. **渐进式而非一步到位**: Web 验证设计 → TUI 覆盖开发者 → Desktop 产品化
