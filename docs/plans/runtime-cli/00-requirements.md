## 背景

**当前状态**：
- `gitim-runtime` binary 只有 HTTP server 形态，listen `127.0.0.1:16868`，无 clap、无 subcommand
- WebUI 可以本地添加 / 管理 agent
- 多节点跨机器互联已经通过 Tailscale + presence-relay 跑通
- Agent provision / burn / per-agent usage / hermes profile isolation 都已落地
- **缺**：agent 无法通过命令行管理 agent —— 所有管理动作只有 WebUI 入口

**触发场景**（用户自述）：
> "我自己有多节点部署的需求，把不同节点部署到不同机器。然后我还有组合测试需求，需要测试不同 harness 配合不同的 provider 的能力。我也不想自己一个个去加 agent。整体的产品诉求就是 ai 做一切，我只在一个窗口发号施令。"

→ 核心需求：**让 agent 能"用嘴部署 agent"**。

---

## Reframe（office-hours 收敛过程的关键转向）

最初的 framing 是 "agent 自主招募 + 多节点编排 + 跨节点 spawn"。经过 forcing questions，scope **砍掉了大约 70%**，落到一个完全不同的形态：

| 初始 framing | 实际形态 |
|---|---|
| Runtime 互相 HTTP 调用 / P2P 跨节点 | Runtime 只 listen 127.0.0.1，**不知道其他 runtime 存在** |
| `gitim-runtime --node alice spawn ...` | **没有 `--node`**，CLI 只操作本机 |
| 命令工具是多节点 control plane | CLI 是 **本机 runtime HTTP API 的 thin shell wrapper** |
| 引入 cluster topology / discovery | 完全不需要 —— topology 不存在于 CLI 视图 |
| 区分常驻 agent vs 日抛 agent | 不区分。所有 agent 一视同仁，user 自己测，不好手动 burn |
| Coordinator 是 protocol first-class | Coordinator 是 **convention only** —— 普通 agent + 注入 system prompt |
| 招募是 message verb | 招募是 **shell tool call**，通过现成的 provider Bash tool |

**关键洞察**：多节点协作**完全通过 GitIM 协议自然 emerge** —— agent 跨机器是 deploy 时的事（user 手工 SSH / pre-provision），跨机器协作通过 `.thread` 消息 + `@mention` 完成。"agent 跑在哪台机器" 是 user 视角的 deployment detail，**不进系统抽象**。

→ Machine 是 invisible，agent 是 first-class。

## 共识 Premises（forcing questions 收敛结果）

### P1 — 单 binary 双模式
`gitim-runtime` binary 同时承担 server 和 CLI：
- 无参数（默认）→ server mode（保持现有行为，零回归）
- 有 subcommand → CLI mode

同一个 binary 既给 human 在 terminal 跑，也给 agent 通过 provider 的 Bash tool shell-out 用。**不另外做 agent-only SDK / tool wrapper**。

**理由**：单 binary 安装物最少；release pipeline 不变；agent shell-out 时 `gitim-runtime list-agents` 比 `gitim runtime list-agents` 更短，对 LLM friendly；用户心智模型已经在这里（"用 gitim-runtime 的某些命令"）。

### P2 — CLI 只操作本机
没有 `--node` 参数，没有远程路由，没有跨节点 awareness。CLI 通过 `127.0.0.1:16868` HTTP 调本机 runtime，仅此而已。

跨节点能力通过 GitIM 协议自身完成（agent 在同一 workspace 互发 `.thread` 消息）。

### P3 — Runtime 不知道其他 runtime 存在
`gitim-runtime` 不引入 Tailscale 概念，不引入多节点概念。任何 cluster awareness 是 **agent 的事**（agent 通过 LLM context 维护它自己看到的世界），不是 runtime 的事。

### P4 — Runtime ID 可查但不进 git
runtime_id 已实现（`~/.gitim/runtime.json`，UUIDv4）。CLI 暴露 `runtime-id` subcommand 让 agent 知道"我跑在哪个 runtime 上"。该信息**永不进 git**。

### P5 — Coordinator 是 convention，通过 system prompt 注入
Coordinator 不是 protocol concept、不是 yaml 字段。它是**一个被 user 信任 + 装了 `gitim-runtime` CLI 作为 Bash tool 的普通 agent**。

"它是 coordinator" 这一事实仅通过 system prompt 表达。worker agent 没装这个 tool 就不能招 —— **不在 prompt 里说"你不能招"，直接不给 tool**。

System prompt 注入复用现有 `users/<handler>.meta.yaml` / `.gitim/me.json.system_prompt` 机制（agent-config-editable 已落地）。CLI 的 `add-agent` 接受 `--system-prompt` 参数。

### P6 — Handler 由 caller 指定
新 agent 的 handler 由 caller 在 `add-agent` 参数里直接给。Runtime 沿用现有 `git fetch origin` + check `users/<handler>.meta.yaml` 存在性的逻辑（github-mode handler 冲突防护已实现），冲突即拒绝。

**不引入中心 handler authority**，不引入自动命名 / 序号分配。Caller 自己负责想名字。

### P7 — Auth = 仅接受本地请求
Runtime 维持 `127.0.0.1:16868` listen（现状），CLI 直接连。不引入 token / signature。**前提**：单机器内的所有 process 视为可信。

### P8 — 协议层零改动
- `users/<handler>.meta.yaml` schema 不动
- `.thread` 文件不动
- `.gitim/me.json` 不动
- 不加 `role` / `node-id` / `capability` 字段
- 不引入新文件类型

## v1 Subcommand 清单

8 个 subcommand，每个都是现有 HTTP endpoint 的 thin wrapper。

| Subcommand | 作用 | 对应 HTTP endpoint |
|---|---|---|
| `add-agent` | provision 新 agent | `POST /workspaces/{slug}/agents` |
| `list-agents` | 列本机 runtime 上所有 agent | `GET /workspaces/{slug}/agents` |
| `burn-agent` | hard / soft delete 指定 agent | `DELETE /workspaces/{slug}/agents/{id}` |
| `update-agent` | 改 agent system_prompt / env / .env | `PATCH /workspaces/{slug}/agents/{id}` |
| `status` | runtime 整体状态（version, uptime, runtime_id, workspace 数, agent 数） | `GET /runtime/health` + 聚合 |
| `workspaces` | 列本机 runtime 服务的 workspace | `GET /workspaces` |
| `runtime-id` | print runtime_id | `GET /runtime/health` |
| `preflight` | check provider CLI 可用性（claude / codex / hermes 等） | `GET /preflight/{provider}` |

### 通用约定

- **输出格式**：默认 JSON（agent friendly，schema 跟 HTTP response 一致）；加 `--pretty` 给 human 友好打印
- **错误处理**：失败 → exit code ≠ 0；人类可读 error 写 stderr；JSON error payload（如有）写 stdout
- **Workspace 选择**：多 workspace 场景下，subcommand 接受 `--workspace <slug>`；省略时如果只有一个 workspace 就默认它，多于一个就报错要求显式
- **HTTP 通信**：直接连 `127.0.0.1:16868`（DEFAULT_PORT 常量）；端口可通过 `--port` 或 `GITIM_RUNTIME_PORT` env override，但 v1 不打文档（仅供测试用）

## Considered Alternatives

### Approach A — `gitim-runtime` 加 subcommand（已选）
单 binary 双模式。

✅ Pros：
- 单个安装物，release pipeline 不变
- Binary name 跟功能直接匹配，发现性强
- Agent shell-out 时命令短

❌ Cons：
- argv parsing 边界要处理好，避免 `gitim-runtime` 零参数误判
- Server 和 CLI 共用一个 binary 增加一些 cognitive load（启动入口需要分流）

### Approach B — 独立 binary `gitim-runtimectl`
Kubectl 风格分离。

❌ 否决理由：多打一个 release target，需多一份 CI 配置；用户多装一个二进制；本项目不是 K8s 量级，分离过度工程。

### Approach C — 加到 `gitim-cli` 作 `gitim runtime <verb>`
统一到 gitim 入口。

❌ 否决理由：gitim-cli 当前只通 daemon（Unix socket），引入 runtime HTTP 通道会混淆两类不同后端；运维语义（管 runtime）和协议语义（发消息）放一起对 LLM caller 不友好。

## Open Questions（留给后续 plan-eng-review / writing-plans）

这些不是 design 决策，是 implementation detail，留给下一阶段：

1. **CLI mode detection 实现细节**：
   - 用 clap 的 subcommand pattern，no-subcommand fallthrough 到 server mode
   - 现有 `gitim-runtime` 启动参数（如果有）怎么映射到 server mode 的 global flags

2. **`status` subcommand 的聚合面**：
   - 单一 HTTP `/runtime/health` 是否包含 agent 列表？还是要并发调多个 endpoint 拼起来？
   - 输出 schema 设计

3. **`add-agent` 参数范围**：
   - 至少需要：`--handler` / `--provider` / `--model` / `--workspace`
   - 可选：`--system-prompt` / `--env KEY=VAL` (repeatable) / `--env-file <path>`
   - System prompt 较长时是否支持 `--system-prompt-file <path>`

4. **`burn-agent` 默认行为**：
   - soft delete 还是 hard delete？
   - 用 `--hard` / `--soft` flag 显式选择

5. **Coordinator standard prompt template**：
   - 是否在 GitIM 内置一段 "你是 coordinator" 的 prompt template？
   - 由 `add-agent --role coordinator` 自动注入？
   - 或者完全交给 user，CLI 不内置 template
   - **倾向**：v1 不内置 template，user 自己写 system prompt；v2 看用户实际使用后再决定

6. **测试策略**：
   - 每个 subcommand 一个集成测试（起 runtime → 跑 CLI → assert HTTP）
   - JSON 输出 schema 用 typed struct 保证
   - 重用现有 `tests/common/mod.rs` 的 daemon 隔离基础设施

7. **文档更新**：
   - 新增 `docs/specs/runtime-cli.md`
   - 更新 CLAUDE.md "Current Orientation"

## 不在 v1 范围（明确 non-goals）

- **`--node <name>` 远程路由** —— P2 说了
- **Tailscale 集成** —— P3 说了
- **Cluster topology / discovery** —— P3 说了
- **Token / API key auth** —— P7 说了；只接受本地请求
- **Agent autonomous recruitment**（agent 自己 reason 出"我要招个人"然后 spawn）—— 这是 LLM 行为层，不是 CLI 层；CLI 只提供 mechanism，policy 由 system prompt 决定
- **Ephemeral / 日抛 agent 抽象** —— Q4 收敛，不区分
- **Auto-burn / cleanup policy** —— 同上
- **Cross-node @mention routing** —— 协议层不动，跨节点自然 emerge
- **Per-node capability registry** —— P8，不进 git
- **CLI 出 server 端的 SSE 订阅** —— v1 拒绝订阅类长连接，CLI 是 one-shot 请求；如要 watch，调 watch loop 是 caller 自己的事（agent 可以循环 `list-agents`）

## 成功标准（v1 ship 时要能做到）

1. **Agent 用 Bash tool 能跑通**：
   ```
   $ gitim-runtime add-agent --workspace test --handler tester-x \
       --provider hermes --model gemini-2.0-flash-exp \
       --system-prompt "You are a test agent."
   {"handler":"tester-x","id":"...","status":"ready"}
   ```
   Coordinator agent 通过 Claude Code 的 Bash tool 调用，能成功 add agent 并拿到 JSON 结果。

2. **8 个 subcommand 全部可用**，每个有集成测试覆盖 happy path + 1 个 error path。

3. **零回归**：现有 WebUI 流程、agent_loop、provider 都不受影响；现有 release pipeline 不变。

4. **输出对 LLM friendly**：JSON schema 稳定，agent 能可靠 parse；error 信息有 `error_code` 字段（沿用现有 HTTP API 约定）。

5. **文档**：`docs/specs/runtime-cli.md` 列出所有 subcommand 用法。
