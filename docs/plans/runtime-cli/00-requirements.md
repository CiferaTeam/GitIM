## 背景

**当前状态**：
- `gitim-runtime` binary 只有 HTTP server 形态，listen `127.0.0.1:16868`，无 clap、无 subcommand
- WebUI 可以本地添加 / 管理 agent
- 多节点跨机器互联已经通过 Tailscale + presence-relay 跑通
- Agent provision / burn / per-agent usage / hermes profile isolation 都已落地
- **缺**：agent 无法通过命令行管理 agent —— 所有管理动作只有 WebUI 入口

**触发场景**（用户自述）：
> "我自己有多节点部署的需求，把不同节点部署到不同机器。然后我还有组合测试需求，需要测试不同 harness 配合不同的 provider 的能力。我也不想自己一个个去加 agent。整体的产品诉求就是 ai 做一切，我只在一个窗口发号施令。"

→ 核心需求：**让 agent 能"用嘴部署 agent"**。

---

## Reframe（office-hours 收敛过程的关键转向）

最初的 framing 是 "agent 自主招募 + 多节点编排 + 跨节点 spawn"。经过 forcing questions，scope **砍掉了大约 70%**，落到一个完全不同的形态：

| 初始 framing | 实际形态 |
|---|---|
| Runtime 互相 HTTP 调用 / P2P 跨节点 | Runtime 只 listen 127.0.0.1，**不知道其他 runtime 存在** |
| `gitim-runtime --node alice spawn ...` | **没有 `--node`**，CLI 只操作本机 |
| 命令工具是多节点 control plane | CLI 是 **本机 runtime HTTP API 的 thin shell wrapper** |
| 引入 cluster topology / discovery | 完全不需要 —— topology 不存在于 CLI 视图 |
| 区分常驻 agent vs 日抛 agent | 不区分。所有 agent 一视同仁，user 自己测，不好手动 burn |
| Coordinator 是 protocol first-class | Coordinator 是 **convention only** —— 普通 agent + 注入 system prompt |
| 招募是 message verb | 招募是 **shell tool call**，通过现成的 provider Bash tool |

**关键洞察**：多节点协作**完全通过 GitIM 协议自然 emerge** —— agent 跨机器是 deploy 时的事（user 手工 SSH / pre-provision），跨机器协作通过 `.thread` 消息 + `@mention` 完成。"agent 跑在哪台机器" 是 user 视角的 deployment detail，**不进系统抽象**。

→ Machine 是 invisible，agent 是 first-class。

## 共识 Premises（forcing questions 收敛结果）

### P1 — 单 binary 双模式
`gitim-runtime` binary 同时承担 server 和 CLI：
- 无参数（默认）→ server mode（保持现有行为，零回归）
- 有 subcommand → CLI mode

同一个 binary 既给 human 在 terminal 跑，也给 agent 通过 provider 的 Bash tool shell-out 用。**不另外做 agent-only SDK / tool wrapper**。

**理由**：单 binary 安装物最少；release pipeline 不变；agent shell-out 时 `gitim-runtime list-agents` 比 `gitim runtime list-agents` 更短，对 LLM friendly；用户心智模型已经在这里（"用 gitim-runtime 的某些命令"）。

### P2 — CLI 只操作本机
没有 `--node` 参数，没有远程路由，没有跨节点 awareness。CLI 通过 `127.0.0.1:16868` HTTP 调本机 runtime，仅此而已。

跨节点能力通过 GitIM 协议自身完成（agent 在同一 workspace 互发 `.thread` 消息）。

### P3 — Runtime 不知道其他 runtime 存在
`gitim-runtime` 不引入 Tailscale 概念，不引入多节点概念。任何 cluster awareness 是 **agent 的事**（agent 通过 LLM context 维护它自己看到的世界），不是 runtime 的事。

### P4 — Runtime ID 可查但不进 git
runtime_id 已实现（`~/.gitim/runtime.json`，UUIDv4）。CLI 暴露 `runtime-id` subcommand 让 agent 知道"我跑在哪个 runtime 上"。该信息**永不进 git**。

### P5 — Coordinator 是 convention，通过 system prompt 注入
Coordinator 不是 protocol concept、不是 yaml 字段。它是**一个被 user 信任 + 装了 `gitim-runtime` CLI 作为 Bash tool 的普通 agent**。

"它是 coordinator" 这一事实仅通过 system prompt 表达。worker agent 没装这个 tool 就不能招 —— **不在 prompt 里说"你不能招"，直接不给 tool**。

System prompt 注入复用现有 `users/<handler>.meta.yaml` / `.gitim/me.json.system_prompt` 机制（agent-config-editable 已落地）。CLI 的 `add-agent` 接受 `--system-prompt` 参数。

### P6 — Handler 由 caller 指定
新 agent 的 handler 由 caller 在 `add-agent` 参数里直接给。Runtime 沿用现有 `git fetch origin` + check `users/<handler>.meta.yaml` 存在性的逻辑（github-mode handler 冲突防护已实现），冲突即拒绝。

**不引入中心 handler authority**，不引入自动命名 / 序号分配。Caller 自己负责想名字。

### P7 — Auth = 仅接受本地请求（**见 §8 修正**）
Runtime 维持 `127.0.0.1:16868` listen（现状），CLI 直接连。不引入 token / signature。**Codex outside voice 指出**：`CorsLayer::permissive()` 让浏览器 driven 的 localhost CSRF 仍是 risk —— "127.0.0.1 only 就安全" 半对。完整 framing 见下方 Architecture decisions §8。本 plan 不修 CORS（pre-existing inherited risk），但 P7 描述要 honest。

### P8 — 协议层零改动
- `users/<handler>.meta.yaml` schema 不动
- `.thread` 文件不动
- `.gitim/me.json` 不动
- 不加 `role` / `node-id` / `capability` 字段
- 不引入新文件类型

## v1 Subcommand 清单

8 个 subcommand，每个都是现有 HTTP endpoint 的 thin wrapper。**注意**：endpoint 名字以下面表格为准（codex 验过 routes 在 `crates/gitim-runtime/src/http.rs:4946-5030`，base prefix 是 `/workspaces/{slug}` for workspace-scoped，否则 root）：

| Subcommand | 作用 | 对应 HTTP endpoint | 备注 |
|---|---|---|---|
| `add-agent` | provision 新 agent | `POST /workspaces/{slug}/agents/add` | 必填：`--handler` `--display-name` `--provider`；Hermes 走 `--llm-provider` + `--llm-model`（不是 `--model`） |
| `list-agents` | 列本机 runtime 上所有 agent | `GET /workspaces/{slug}/agents` | 默认返 redacted 视图（见 Architecture decisions §3） |
| `burn-agent` | 公开 ritual burn（默认） / hard remove（`--hard`） | `POST /workspaces/{slug}/agents/burn` 或 `/agents/remove` | `burn` 走广播 ritual，`--hard` 走 hard delete 不公告 |
| `update-agent` | 改 agent system_prompt / env / .env / introduction | `PATCH /workspaces/{slug}/agents/{id}` | |
| `status` | runtime 整体状态（version, uptime, runtime_id, workspace 数, agent 数） | `GET /health` + 多 workspace 聚合 | 注意：是 `/health` 不是 `/runtime/health` |
| `workspaces` | 列本机 runtime 服务的 workspace | `GET /workspaces` | |
| `runtime-id` | print runtime_id | `GET /health`（runtime_id 是 /health 响应字段） | |
| `preflight` | check provider CLI 可用性（claude / codex / hermes 等） | `GET /preflight/{provider}` | |

### 通用约定

- **输出格式**：默认 JSON；加 `--pretty` 给 human 友好打印。**JSON schema 是 CLI 自己定义的 typed wire structs**（见 Architecture decisions §2），不裸 expose runtime 内部 `AgentInfo`
- **错误处理**：靠 **响应 body 里的 `error_code` 字段** classify，不靠 HTTP status（见 §1）；exit code 分档（见 §4）
- **Workspace 选择**：subcommand 接受 `--workspace <slug>`；省略时如果只有一个 workspace 就默认它，多于一个就报错要求显式
- **HTTP 通信**：CLI 启动时**先读 `~/.gitim/runtime.json` 的 `listen_port` 字段**（见 §5），fallback 到 `127.0.0.1:16868`（DEFAULT_PORT 常量）。可被 `--port` flag 或 `GITIM_RUNTIME_PORT` env override，用于测试

## Architecture Decisions（plan-eng-review + codex outside voice 沉淀）

这一节是 design review 输出，**必须**落地在实现里。每一条都标了为什么以及 codex 验证的代码位置。

### §1 — Error classification 靠 body `error_code`，不靠 HTTP status

**Why**: runtime 多处 failure 路径返回 **HTTP 200 + `{ok:false, error_code:"..."}`**。例：`handler_conflict` at `http.rs:2147-2152`、`hermes_not_setup` at `2361-2369`、`provision_agent failed` at `2589-2594`、`agent not found` at `2944`。HTTP status 在 runtime contract 里不可信。

**How**:
- CLI 解析 response body 的 `error_code` 字段决定行为
- `error_code` 已知值（来自 grep）：`handler_conflict` / `hermes_not_setup` / `hermes_profile_create_failed` / `token_lacks_repo_access` / `invalid_token` / `insufficient_scope` 等
- 这些归为 **permanent error，agent 不该 retry**
- 网络层错误（connect refused 等）、parse fail 归为 **transient / config error**

### §2 — CLI 自己定义 typed wire DTO，不复用 runtime lib structs

**Why**: codex 验证 lib structs 大部分 private —— `ErrorBody` (http.rs:107)、`AgentsListResponse` (193)、`AgentAddResponse` (207) 都是 `struct`（private）。`AgentInfo` 是 `pub` 但只 derive `Serialize`，没 `Deserialize`（http.rs:226-227）。CLI 想 typed 必须自己定义 wire structs，或者 parse `serde_json::Value`。

**How**: CLI 模块定义自己的 `AgentView`（redacted 公开视图）、`AgentDetail`（完整含 secrets，要 `--detailed` flag）、`ErrorResponse`、`AddAgentResponse` 等 wire DTOs，全部 `Serialize + Deserialize`。**修改不动 runtime lib**。

### §3 — `list-agents` 默认 redacted view（CLI-safe）

**Why**: `AgentInfo` at `http.rs:226-269` 含敏感字段：`repo_path`（absolute local path 泄露机器布局）、`system_prompt`（私有 operational instructions）、`env`（HashMap，可能含 API key 类）、`session_usage` / `usage_summary`（业务遥测）。Default `list-agents` 给 agent / human 看时**不能 dump 这些**。

**How**:
- `AgentView`（默认 schema）：`id`, `handler`, `display_name`, `status`, `provider`, `model`, `messages_processed`, `last_activity`
- `AgentDetail`（`--detailed` flag）：加 `system_prompt`, `repo_path`, `env`（**仍然 redact 已知 secret-shaped keys，如 含 "KEY" / "TOKEN" / "SECRET" 子串**），`session_usage`, `usage_summary`
- `agent-get` subcommand（如未来加）也走 detailed 视图
- 安全 default：能少给就少给，agent 显式要才给

### §4 — Exit code 分档（基于 §1 的 error_code）

| Code | 语义 | Agent 该做什么 |
|---|---|---|
| 0 | success | continue |
| 1 | CLI 内部错（argv / connect refused / parse fail / config error） | retry-after-fix-config |
| 2 | HTTP 响应 body 含 `error_code`（permanent error） | **不要 retry**，改输入 |
| 3 | HTTP 5xx 或 transient runtime 错（empty body / timeout） | 可 retry with backoff |

**注意**：因为 §1，code 2 主要靠 body error_code 判断，不是 HTTP 4xx。

### §5 — Runtime 持久化 `listen_port` 到 `runtime.json`

**Why**: 当前 `~/.gitim/runtime.json` 只存 `runtime_id` + `workspaces`（`user_config.rs:11-19`）；`listen_port` 只在 in-memory state（`bin/runtime.rs:143-146`）。非 default port 时 CLI 跟丢。

**How**:
- `UserConfig` schema 增加 `listen_port: Option<u16>`
- Runtime 启动 bind 成功后写回 `runtime.json`
- CLI 启动时读 runtime.json 取 port，fallback default
- 多 runtime 在同机器同 user：不在 v1 scope（user 用单一 runtime 模式）
- **runtime_id 一样不进 git**（已是现状）

### §6 — `check_env()` preflight 移到 server-only 分支

**Why**: `bin/runtime.rs:27-31` 在 mode dispatch **之前**跑 `check_env()`，CLI 模式也会 fail。但 binary 版本不齐对 `gitim-runtime list-agents` 没影响 —— already-running runtime 完全能回应。

**How**: 重构 main：
```
main() → parse argv (clap) → 
  if subcommand → run cli (NO check_env)
  else → check_env + run server
```

### §7 — Legacy positional-arg agent mode retire

**Why**: `bin/runtime.rs:46-83` 的 dead code，grep 全 codebase 无 caller，跟 clap subcommand 是 hard conflict。

**How**: 删除 legacy 分支，clap subcommand 全占据 positional arg 空间。`bin/runtime.rs` 重写为 clap `Parser` + `Option<Command>` pattern。

### §8 — CORS permissive 是 inherited known risk，**不在本 plan 修**

**Why**: `http.rs:5030` 的 `CorsLayer::permissive()` 让浏览器从任何 origin 都能 POST 到 127.0.0.1:16868。**P7 描述的 "127.0.0.1 only 就安全" 实际半对** —— 浏览器 driven 的 localhost CSRF 仍可能。CLAUDE.md "Current Orientation" 已 flag 这个 known issue（"update-and-restart endpoint 继承 permissive CORS, 整站 CSRF 是 known risk"）。

**How**: 修正 P7 的 framing（见下），但**不在本 plan scope 修**。CLI feature 不引入新 attack surface，只是 ride 现有 HTTP。修 CORS 是独立 PR。

---

### P7 修正版

> Runtime 维持 `127.0.0.1:16868` listen（现状）。CLI 直接连本机 HTTP。**Auth posture = 继承现有 runtime 安全模型**（无 auth + permissive CORS），不引入额外 auth 层。**Known risk**: 浏览器 driven 的 localhost CSRF 是 pre-existing 风险（CLAUDE.md 已 flag），不在本 plan scope 解决。

## Considered Alternatives

### Approach A — `gitim-runtime` 加 subcommand（已选）
单 binary 双模式。

✅ Pros：
- 单个安装物，release pipeline 不变
- Binary name 跟功能直接匹配，发现性强
- Agent shell-out 时命令短

❌ Cons：
- argv parsing 边界要处理好，避免 `gitim-runtime` 零参数误判
- Server 和 CLI 共用一个 binary 增加一些 cognitive load（启动入口需要分流）

### Approach B — 独立 binary `gitim-runtimectl`
Kubectl 风格分离。

❌ 否决理由：多打一个 release target，需多一份 CI 配置；用户多装一个二进制；本项目不是 K8s 量级，分离过度工程。

### Approach C — 加到 `gitim-cli` 作 `gitim runtime <verb>`
统一到 gitim 入口。

❌ 否决理由：gitim-cli 当前只通 daemon（Unix socket），引入 runtime HTTP 通道会混淆两类不同后端；运维语义（管 runtime）和协议语义（发消息）放一起对 LLM caller 不友好。

## Open Questions（留给后续 plan-eng-review / writing-plans）

这些不是 design 决策，是 implementation detail，留给下一阶段：

1. **CLI mode detection 实现细节**：
   - 用 clap 的 subcommand pattern，no-subcommand fallthrough 到 server mode
   - 现有 `gitim-runtime` 启动参数（如果有）怎么映射到 server mode 的 global flags

2. **`status` subcommand 的聚合面**：
   - 单一 HTTP `/runtime/health` 是否包含 agent 列表？还是要并发调多个 endpoint 拼起来？
   - 输出 schema 设计

3. **`add-agent` 参数范围**：
   - 至少需要：`--handler` / `--provider` / `--model` / `--workspace`
   - 可选：`--system-prompt` / `--env KEY=VAL` (repeatable) / `--env-file <path>`
   - System prompt 较长时是否支持 `--system-prompt-file <path>`

4. **`burn-agent` 默认行为**：
   - soft delete 还是 hard delete？
   - 用 `--hard` / `--soft` flag 显式选择

5. **Coordinator standard prompt template**：
   - 是否在 GitIM 内置一段 "你是 coordinator" 的 prompt template？
   - 由 `add-agent --role coordinator` 自动注入？
   - 或者完全交给 user，CLI 不内置 template
   - **倾向**：v1 不内置 template，user 自己写 system prompt；v2 看用户实际使用后再决定

6. **测试策略（hybrid 三层，integrated codex push back）**：
   - **Layer A — clap parse unit tests**：`#[test]` direct call `clap::Parser` API，cover argv edge cases：no args、`--version`、`--port 0`、`--port` 缺 value、`--daemon --port`、unknown subcommand、旧 positional args 必须 error
   - **Layer B — handler integration tests**：复用 `tests/common/mod.rs` daemon 隔离（`ensure_daemon_in_path` + `GITIM_LOG_DIR`），起真 runtime，直接 call `cli::add_agent::run(args)` 等 handler functions。每 subcommand 1 happy + 1 error path
   - **Layer C — binary subprocess smoke test**：1-2 个 `Command::new("gitim-runtime")` 实际 fork binary 的 smoke，cover：legacy mode 退役后旧用法 exit non-zero、`--version` 输出、`gitim-runtime` no args 仍走 server mode（spawned process 启动后 ping `/health` 然后 kill 即可）
   - JSON 输出 schema 用 CLI 自定义 typed wire struct（见 Architecture decisions §2）保证

7. **文档更新**：
   - 新增 `docs/specs/runtime-cli.md`
   - 更新 CLAUDE.md "Current Orientation"

## 不在 v1 范围（明确 non-goals）

- **`--node <name>` 远程路由** —— P2 说了
- **Tailscale 集成** —— P3 说了
- **Cluster topology / discovery** —— P3 说了
- **Token / API key auth** —— P7 说了；只接受本地请求
- **Agent autonomous recruitment**（agent 自己 reason 出"我要招个人"然后 spawn）—— 这是 LLM 行为层，不是 CLI 层；CLI 只提供 mechanism，policy 由 system prompt 决定
- **Ephemeral / 日抛 agent 抽象** —— Q4 收敛，不区分
- **Auto-burn / cleanup policy** —— 同上
- **Cross-node @mention routing** —— 协议层不动，跨节点自然 emerge
- **Per-node capability registry** —— P8，不进 git
- **CLI 出 server 端的 SSE 订阅** —— v1 拒绝订阅类长连接，CLI 是 one-shot 请求；如要 watch，调 watch loop 是 caller 自己的事（agent 可以循环 `list-agents`）

## 成功标准（v1 ship 时要能做到）

1. **Agent 用 Bash tool 能跑通**：
   ```
   # 普通 provider（claude / codex / openai）
   $ gitim-runtime add-agent --workspace test --handler tester-x \
       --display-name "Tester X" --provider claude --model claude-opus-4-7 \
       --system-prompt "You are a test agent."
   {"ok":true,"id":"..."}

   # Hermes 走 --llm-provider + --llm-model（不是 --model）
   $ gitim-runtime add-agent --workspace test --handler tester-y \
       --display-name "Tester Y" --provider hermes \
       --llm-provider gemini --llm-model gemini-2.0-flash-exp \
       --system-prompt "You are a test agent."
   {"ok":true,"id":"..."}
   ```
   Coordinator agent 通过 Claude Code 的 Bash tool 调用，能成功 add agent 并拿到 JSON 结果。

2. **8 个 subcommand 全部可用**，每个有集成测试覆盖 happy path + 1 个 error path。

3. **零回归**：现有 WebUI 流程、agent_loop、provider 都不受影响；现有 release pipeline 不变。

4. **输出对 LLM friendly**：JSON schema 稳定，agent 能可靠 parse；error 信息靠 body 里 `error_code` 字段 classify（Architecture decisions §1）；exit code 0/1/2/3 分档（§4）让 agent 能 reason out retry-able vs permanent。

6. **默认 `list-agents` 不泄露敏感字段**（repo_path / system_prompt / env / session_usage 都不在默认输出）；要看完整细节走 `--detailed` flag（Architecture decisions §3）。

7. **CLI 模式下 `check_env()` 不跑**（不被 binary version mismatch 阻挡），server 模式仍跑（Architecture decisions §6）。

5. **文档**：`docs/specs/runtime-cli.md` 列出所有 subcommand 用法。
