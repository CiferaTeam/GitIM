## 问题陈述

phase0(`feature/daemon-web-phase0`)做了一次"在浏览器里跑 GitIM"的探索:

- 加 `gitim-wasm` crate,把 `gitim-core`/`gitim-sync` 的纯函数桥到 JS
- 给 `gitim-sync` 的 I/O 模块做 cfg 门控,workspace 能编译到 wasm32
- webui-v2 引入 `Backend` interface(HttpBackend / LocalBackend)
- 加了 531 行 TS 实现的 daemon handler、TS 重写的 parser/formatter/conflict、isomorphic-git + lightning-fs 套件
- 浏览器 worker 跑一份 mini daemon,push-first sync 到远端 git 仓库

**phase0 的模型是对的(浏览器作为 git peer 在技术上可行),但实现走偏**:把 Rust 现成的 handler 决策逻辑用 TS 重写了一遍,而 gitim-wasm 暴露的纯函数 TS 反而没用。结果是两份代码,daemon 一改、浏览器侧立刻漂移。card archive 系统(2026-04 进 main)是直接证据 — phase0 的 handlers.ts 完全不知道 cards 存在。

而且 phase0 的产品定位被框成了"移动端人类聊天客户端"。这个定位是错的。

## Reframe — 两个部署模式

phase0 的"浏览器跑 daemon"本质上不是 mobile feature,而是一种 **部署模式**(deployment mode)。一旦它能跑,**任何带浏览器的设备**都可以用同一个模式 — 包括桌面。

GitIM 因此应有两种部署模式并存:

### Mode 1 — Daemon 模式(当前)

- 用户设备上有一个常驻的 `gitim-runtime`/`gitim-daemon` 进程
- 进程持有 git 仓库的工作树,通过 git2/git CLI 做 pull/push,文件系统 + SQLite 做持久化
- 用户**可以创建并管理 agent**(agent 跑在同一台机器上,共享 daemon 的 IM 资源)
- WebUI 通过 Unix socket / HTTP 对话 daemon
- 用例:开发者本机、agent 工作站、长期挂机的设备

### Mode 2 — Browser-only 模式(新)

- 设备上**没有任何 GitIM 进程**
- 浏览器打开 webui → 用户粘 GitHub PAT → 浏览器 WASM 模块 + isomorphic-git + IndexedDB 自己 clone 仓库,自己 pull/push
- 浏览器本身就是这个 handler 的 git peer
- 用户**只能作为人类参与**(发消息、读消息、看 cards/agents 现状)— **不能创建/管理 agent**(agent 需要持续运行进程,关 tab 即下线)
- WebUI 通过 Web Worker RPC 对话浏览器内的 mini runtime
- 用例:手机、临时设备、刚来想试试的用户、协作里"只参与对话不挂机"的角色

### 关键洞察

两个模式**共享同一份 WebUI**。区别只在 Backend 实现:

```
                  webui-v2 (React 组件 + 路由,不变)
                            │
                            ▼
                    Backend interface  ← phase0 已设计
                   ╱                 ╲
            HttpBackend           BrowserBackend
                │                       │
        Unix socket / HTTP       Web Worker RPC
                │                       │
       gitim-daemon (native)     mini runtime in WASM
                │                       │
        local git2 + fs         isomorphic-git + IndexedDB
                ╲                       ╱
                 ╲                     ╱
                  ╲                   ╱
                   远端 git 仓库(GitHub / Gitea / GitLab)
                       ↑ 单一 source of truth ↑
```

两边 push/pull 同一个 git remote,通过 git 自然同步。这正是 GitIM 一开始就该有的形态 — phase0 实现错了,但模型是对的。

## 模式边界

| 能力 | Daemon 模式 | Browser-only 模式 |
|---|---|---|
| 收发消息(channel + DM) | ✅ | ✅ |
| 读 cards / agents 状态 | ✅ | ✅ |
| 创建 / 修改自己的 cards | ✅ | ✅(对应文件由浏览器写并 push) |
| 修改 agent 创建 / 持有的 cards | ✅(daemon 协调) | ⚠️ 受 daemon 同款权限规则约束(`creator-or-assignee`),原则上能改;但 agent 不在场时的协作语义需 v1 设计 |
| 创建 / 管理 agent | ✅ | ❌(agent 需常驻进程) |
| 接收实时事件(SSE 推送) | ✅ | ⚠️ 由浏览器 polling 模拟,默认 cadence ≥ 7s |
| 离线写入(关网仍可发送) | ✅ | ✅(IndexedDB 暂存,在线时 push) |
| 长期挂机 / 守护进程能力 | ✅ | ❌ |
| Workspace github/local provider | ✅ 双支持 | ⚠️ github only(local provider 需要本地 fs) |

**Mode 2 不是 Mode 1 的 subset,而是 sibling**:同一个用户可以两端同时登录(电脑用 Daemon、手机用 Browser-only),作为同一个 handler 在 git 上是同一个身份。冲突由 sync_loop 的 push-first 策略解决(沿用 phase0 算法)。

## 信任边界(Browser-only mode)

Mode 2 的安全姿态比 Mode 1 弱,需明确的有 3 点:

1. **PAT 客户端持有,XSS 即权限失守** — 浏览器内的 GitHub PAT 等同于 git 仓库写权限;任何注入到 webui-v2 的 JS(自身 bug、第三方依赖供应链、CDN 劫持)都能拿到。Mode 2 应**显式只接受 fine-grained PAT,且仅授予 single-repo 权限**;不允许使用 `repo` scope 经典 token
2. **isomorphic-git / lightning-fs 等浏览器侧依赖是 Mode 2 独有的攻击面** — 必须做 SBOM 锁定 + 定期审计;不引入需要外部 CDN 的依赖
3. **Mode 2 用户不能装 agent 还有第二层意思** — agent 是 long-running、token-bearing 的进程,在浏览器内即便技术能跑,token 暴露面也无法接受;产品上明确禁止

详细 threat model + 缓解措施见 Stage B/C 实施前的安全 design(独立 doc)。

## 浏览器→GitHub 通信假设

phase0 用 isomorphic-git + GitHub HTTPS。**未验证 GitHub `https://github.com/...` 直连是否对所有浏览器可用** — 历史上 isomorphic-git 文档建议接 CORS proxy(`@isomorphic-git/cors-proxy` 或 cors.isomorphic-git.org);如自己跑 cors proxy,需评估其在 Mode 2 部署形态下的来源(自托管 / 公共)。

Stage E 部署形态决策必须 cover 这一项;Stage C 实施前要 spike 一次"是否需要 cors proxy 中介"。

## 命名澄清

现有"guest mode"在 daemon 里指**写权限受限的 PAT**(见 `archive_card`/`unarchive_card` 的 guest_mode 写 denylist)。这是 daemon 内部的权限模型,与本设计中 Mode 2 是**不同概念**,不能混用名字。

本 doc 暂用:

- **Daemon mode** — 装 runtime
- **Browser-only mode** — 不装 runtime,纯浏览器

最终命名(产品语)在 v1 实施前确定。候选词:Resident/Ephemeral、Studio/Reader、Full/Lite。

## 架构前提:gitim-handlers crate(B 路线)

**Mode 2 要可持续维护,必须先做 handler 共享层重构。** 这是 reframe 的硬依赖,不是可选项。

phase0 选择 TS 重写 handler 的根因是 `gitim-wasm` 不能依赖 `gitim-daemon`(daemon 拖着 tokio + git2)。而 handler 决策逻辑当时混在 daemon I/O 里,没法抽出来给 wasm 用 — 退而求其次只好 TS 重写。

正确做法:把 handler 的"决策"部分拆到独立 crate,native 与 wasm 都从这个 crate 取。

```
crates/
├── gitim-core/        类型 + parser + formatter + validator(不变)
├── gitim-handlers/    ★ 新建 — pure decision functions
│   └── src/
│       ├── snapshot.rs    Snapshot<'a> { thread_content, channel_meta, user_list, ts, ... }
│       ├── effect.rs      Effect { AppendFile / WriteFile / GitMv / Commit }
│       ├── send.rs        pub fn handle_send(snap, req) -> Result<Vec<Effect>, Error>
│       ├── archive_card.rs ...
│       └── ... (~23 handlers)
├── gitim-daemon/      → thin native runtime:I/O + RPC + AppState 管理
└── gitim-wasm/        → 把 gitim-handlers 的函数桥到 JS
```

native handler 退化为统一形态:

```rust
async fn handle_send(state: AppState, req: SendRequest) -> Response {
    let snap = build_snapshot(&state, &req).await?;     // I/O,5 行
    let effects = gitim_handlers::handle_send(&snap, &req)?; // 1 行
    apply_effects(&state.git_storage, effects).await?;  // I/O,5 行
    Response::ok()
}
```

WASM runtime 形态完全对偶:

```ts
async function send(req) {
  const snap = await buildSnapshot(req);              // IndexedDB 读
  const effects = wasm.handle_send(snap, req);        // wasm,1 行
  await applyEffects(effects);                        // IndexedDB 写 + isomorphic-git commit
}
```

**核心收益:**

1. handler 决策只有一份代码,daemon 改 = wasm 改 = mobile/browser-only 改
2. native daemon handler 从 100+ 行缩到 ~10 行
3. Snapshot/Effect 把"纯/不纯"边界画死,纯函数好测(无需 mock fs/git)
4. 新加 feature = `gitim-handlers` 新 fn,两个 runtime 自动跟上,**漂移结构性消除**

**Snapshot 草图(Stage B 起点,非最终):**

```rust
pub struct Snapshot<'a> {
    pub thread_content: Option<&'a str>,        // 当前 channel/DM 的 .thread 文件
    pub channel_meta:   Option<&'a ChannelMeta>,// channels/<x>.meta.yaml
    pub archive_channel_meta_exists: bool,      // archive 目录是否有同名(决定能否写)
    pub user_list:      &'a [String],           // users/ 目录全量
    pub timestamp:      &'a str,                // 注入,纯函数不取时钟
    pub me:             &'a Handler,            // 当前用户
}
```

字段会随 handler 增多而扩展(card / board / dm 各有自己的 pre-load 需求);Stage B 第一步就是把 Snapshot **完整化**到能 cover 所有 23 个 handler。

**Effect 分组语义:**

- `AppendFile` / `WriteFile` / `GitMv` 是**改动声明**,本身不 commit
- `Commit` 必须列出 **paths**(本次要打包提交的所有路径)+ msg + author
- 一次 handler 调用返回的 `Vec<Effect>` 通常以**一个或多个 Commit 收尾**;runtime 按顺序应用,每个 Commit 是 git 上的原子单位
- 如果 handler 只读不写(如 `read_thread`),返回 `Vec::new()`,response 由其他通道返回

**核心代价:**

1. 需要先设计好 `Snapshot`(caller 与 handler 的契约) — 设计错了得返工。Stage B 的 hold point 即此
2. **当前 `AppState` 不能直接喂给 pure handler** — 它含 `Arc<Mutex<...>>`、`Arc<GitStorage>`、`PathBuf` 等不可序列化、不可跨 wasm 边界的字段。**Snapshot 必须在 handler 视角下完全替代 AppState**(不是补充),这是设计上的硬要求。AppState 内的内存缓存(如 user_list、channel_meta 缓存)按需在 build_snapshot 阶段读出来传入
3. **23 个 handler 全部迁移**(每个变简单,但工作量在那里)。增量 PR(handler-by-handler)避免一次性大动土
4. 触动 daemon 主体,但**不需要全局冻结**:每个 handler 独立 PR,与 in-flight 的小改动并行,通过定期 sync(每周一次)收敛分歧

## 落地节奏

> 命名约定:为避免与 `feature/daemon-web-phase0` 分支重名,本 doc 内部一律用 **Stage** 而非 Phase。

### Stage A —— 设计完成(本文档)
产出双模式 + B 路线总体设计,达成共识。

### Stage B — handler 共享层重构(独立 session)
**不**全局冻结 main。采用 handler-by-handler 增量迁移,每个 handler 一个 PR、独立可 review/可回滚:

1. 先取 1-2 个代表性 handler(`handle_send`、`handle_join_channel`)做原型,锁定 `Snapshot` / `Effect` / `HandlerError` 类型
2. 建 `gitim-handlers` crate,迁这两个 handler,native daemon 切到新形态,跑通所有现有测试
3. 锁定接口后,**剩余 21 个 handler 按"组"分批 PR**(message / channel / board / card / dm / onboard 几大组),与 main 上 in-flight 的小改动并行(每周 sync 一次到 main)
4. `gitim-wasm` 全量 export `gitim-handlers` 函数
5. main 与 Stage B worktree 接近收敛后,做最后一次 rebase + merge

预估总历时 2-4 周(无单点冻结,但 handler 完全迁移完成与 main 并跑要分批)。Snapshot 设计完成前不开始 handler 大批量迁移 — 这是关键 hold point。

### Stage C — Browser-only runtime 重写
phase0 的 webui-v2/src/daemon-web/ 全部清空,按以下结构重写(沿用 phase0 已验证的脚手架):

- `worker.ts` — Web Worker entry,RPC dispatch(保留)
- `runtime.ts` — 包浏览器 IndexedDB 状态(取代 state.ts)
- `git.ts` — isomorphic-git 操作(保留)
- `storage.ts` — lightning-fs IndexedDB(保留)
- `sync.ts` — push-first sync loop(保留算法,改用 wasm conflict 函数)
- `handlers.ts` — 每个 handler ≤ 15 行,做 buildSnapshot → wasm 调用 → applyEffects(完全重写)
- ❌ parser.ts / formatter.ts / conflict.ts — **删除**(改用 wasm 调用)

### Stage D — webui-v2 模式接入
- `Backend` interface 加一个 `mode` 字段,组件按 mode 隐藏不该出现的功能(agent 管理、新建 agent)
- LocalSetup 表单(phase0 已写)沿用,改名为 BrowserSetup
- 路由层补 mode-aware 默认页(Browser-only 默认 /chat)
- 设置页面给 Browser-only mode 加 token 重新输入入口
- **命名最终化截止 Stage D 启动前**:Daemon mode / Browser-only mode 等内部叫法在此前定型

### Stage E — 部署形态决策(独立 design 阻塞 Stage C)
Browser-only mode 的实际部署面 — 自托管 / gitim.io / 纯静态 + GitHub Pages — 与产品/运营独立讨论,**但 Stage C 实施前必须有结论**:CORS proxy / token 管理 / 域名等会决定 runtime.ts 与 git.ts 的具体写法。本 doc 不强行定夺。

## 待解开放问题

(全部留给后续 session,本 doc 不强行回答)

1. **PAT 在 Browser-only 浏览器里存哪?** localStorage / IndexedDB / sessionStorage / OPFS,各自的 XSS / 多账号切换 / 跨设备恢复风险
2. **Backend 模式怎么探测?** 启动时 probe `/health` ?URL 区分?build-time flag?用户首次访问时 onboarding 流程
3. **AppState 里的 in-memory cache 具体怎么搬到 build_snapshot?** 主文已定方向(读出来传入),具体每个缓存的访问模式在 Stage B 验证
4. **Daemon mode 与 Browser-only mode 同时同 handler 在线**(电脑桌面 + 手机 web 同账号)— sync_loop 的 push-first 算法是否能 cover 所有冲突 case,还是要加 last-writer-wins 类策略
5. **Browser-only 模式下 SSE / 长轮询的实际可用性** — 浏览器 tab 后台时的节流影响
6. **Workspace local provider 在 Browser-only 模式下不可用** — UI 怎么提示
7. **gitim-handlers 的 AppState 接口** — 用 trait 还是 enum?是否需要 async handler?
8. **错误模型** — handler 返回 Result<Vec<Effect>, HandlerError>,HandlerError 怎么分类(用户错 / 系统错 / 冲突 / git push 失败)
9. **Stage B 期间 main 紧急修复合并冲突** — handler-by-handler 迁移过程中,main 上某个还没迁的 handler 改动如何 sync 到迁移分支
10. **agent 不在场时 cards 的协作语义** — Browser-only mode 用户改 agent 创建的 card 时,agent 下次上线如何感知/对账;v1 设计前不允许 Mode 2 写 agent-owned cards 还是允许 last-writer-wins

## 成功标准

本设计的成功标准是 **3 个月内能演示**:

1. 同一个用户在桌面用 Daemon 模式 + 在手机用 Browser-only 模式同时在线
2. 桌面发消息,手机 7 秒内收到;手机发消息,桌面 3 秒内收到
3. 桌面 agent 写出一张 card,手机能看到该 card(不能修改)
4. handler 加新 feature(随便举一个,如 reaction)只改 `gitim-handlers` 的一个文件,daemon + 浏览器都能用
