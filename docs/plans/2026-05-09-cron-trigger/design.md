# Cron Trigger 设计

## 问题

gitim agents 当前是完全被动的 — 只在被 @-mention 或收到 DM 时唤醒。需要一种机制让 agent 在固定时间点自主行动，例如"每周一 9 点写周报"、"每天上班前扫一下 GitHub PR"、"每小时刷一次外部 API 信号"。

## 核心 framing：cron = 协议一等公民 + 复用消息机制

Cron 是协议级别的新目录 `crons/`，但触发行为完全复用 thread + 消息 + agent_loop 现有路径。到点时 daemon 假装自己叫 `@system` 把一条 prompt 写进对应 thread 文件，目标 agent 通过现有 poller 路径醒来处理。

这保留了 gitim 的核心 invariant —— **agent 行为通过 thread 消息流动** —— 而不是开外挂破坏它。

## 协议层：文件布局

```
<workspace>/
  crons/
    weekly-report/
      spec.yaml                       # cron 表达式 + target + prompt 模板
      2026-05-11T09-00-00Z.thread     # 周一 9 点这一次执行
      2026-05-18T09-00-00Z.thread     # 下周一这一次执行
    daily-standup/
      spec.yaml
      2026-05-11T09-30-00Z.thread

  archive/                            # top-level，与 channel-archive 一致
    crons/
      old-job/
        spec.yaml
        2026-04-...thread
```

每个 cron job 一个文件夹。`spec.yaml` 是配置；每次执行一个独立 `.thread` 文件，文件名嵌**理论 fire 时刻**（不是实际写入时刻 — 见下方 Idempotency），ISO8601 with `:` → `-` 兼容文件系统。

软删除走 `git mv crons/<name>/ → archive/crons/<name>/`，与 [crates/gitim-daemon/src/handlers/channel.rs:221](crates/gitim-daemon/src/handlers/channel.rs:221) 的 channel-archive 实现保持一致（top-level `archive/`，不是 `crons/.archive/`）。

### 为什么"每次执行一个 thread"，而不是"一个 job 一个长 thread"
- 每次执行是独立上下文（fire → start → done），不该混在一个长 thread
- 长 thread 永远涨，renumber 越来越贵
- "重跑某次" / "查那个周一发生了什么" — 独立文件干净
- 前端展示成本 0：`ls crons/<job>/` 看 job 历史；glob `crons/*/2026-*.thread` 看时间线，文件名直接是排序键

代价：git 仓库会涨。但 cron 日志本来就是高 churn，可接受。v2 加 prune 策略（保留最近 N 次）。

### spec.yaml 字段（最终以 Rust 类型为准，参考 `feedback_code_is_contract.md`）
- `version: 1` — schema 版本，forward-compat
- `schedule`: 标准 5-field cron + alias（`@daily/@hourly/@weekly`）
- `timezone`: IANA 字符串（`America/Los_Angeles`），缺省 UTC
- `target`: 目标 handler（具体值，如 `@analyst`）。**create-time 已解析**：CLI 拿到 `--target @self` 时立刻替换为创建者实际 handler，spec 不存占位符（避免不同 clone 解释不一）
- `prompt`: 触发时写进 thread 的 body
- `enabled`: bool，默认 true（disable 用）
- `created_by`, `created_at`

不存 mutable runtime 状态（如 `last_missed_at`）— 错过的 fire 由 calendar UI 在读取时实时计算（`spec.schedule × 时间窗 - 实际 .thread 文件 = 缺失`），避免 daemon 写出大量 trivial commit 污染 git log。

具体 Rust 类型落 `crates/gitim-core/src/types/cron.rs`，配解析 + 校验 + 默认值 + extra flatten + tests。

### 为什么不用现有 `users/<handler>.meta.yaml`
- UserMeta 是身份字段，不是行为字段
- 一个 agent 多个 cron 时 schema 难看
- 跨 agent 调度（一个 agent 给另一个加 cron）会破坏 ownership 语义
- 历史 fire 记录没有合适的协议位置

## Runtime / Daemon 层：cron 引擎

### 引擎位置：daemon 内部
新模块 `crates/gitim-daemon/src/cron_engine.rs`，在 `lifecycle.rs` 启动序列里 spawn 一个 tokio 任务（与 sync_loop 平行），每分钟唤醒一次。

### 关键算法（含三个不变式）

```
loop every 60s:
  self_handler = read .gitim/me.json

  for each spec_dir in crons/*/ (skip archive/):
    spec = parse spec_dir/spec.yaml
    if !spec.enabled: continue

    # ① Ownership filter（多 clone 防重复 fire）
    if spec.target != self_handler: continue

    # ② Idempotency（fs is truth；新建时 last_fire = created_at）
    last_fire = max(parse ts from spec_dir/<ts>.thread filenames)
                || spec.created_at
    next_due = croner::next_fire_after(spec.schedule, last_fire, spec.timezone)

    if next_due <= now:
      # ③ 文件名嵌的是理论 fire 时刻，不是实际写入时刻
      fire(spec, theoretical_ts = next_due)
```

`fire()` 内部：
1. 拿 `state.commit_lock.lock()`（参考 [crates/gitim-daemon/src/handlers/send.rs:51](crates/gitim-daemon/src/handlers/send.rs:51) 同款 pattern）
2. 写 `crons/<name>/<theoretical_ts>.thread`，首行：`[L1][@system][<ts>] cron(<name>): <prompt>`
3. `git add` + `git commit`（commit author = `@system`，email = workspace owner github_email）
4. 释放 lock
5. sync_loop 异步 push（不需特殊路径）
6. 目标 handler 自己的 runtime 通过 poller 拿到 ChannelChange，`format_changes_as_prompt` 生成 prompt，agent 醒来处理

### 三个不变式（出 bug 时回这里查）
- **Ownership**：`spec.target` 必须 == 本 clone handler。多 clone 因此各自只 fire 自己 owned 的 cron，永不重复
- **Idempotency**：`<theoretical_ts>.thread` 文件存在 == 已 fire。daemon 重启 / 时钟漂移 / 扫描在分钟边界附近 — 重复扫描看到同样的 next_due，但写出文件名相同 → fs 拒绝 / engine 跳过
- **Bootstrap**：fresh spec 的 `last_fire = spec.created_at`。"创建即 fire"不会发生（除非 created_at 之后到 now 之间有 schedule 匹配，那是用户故意为之）

### 选库：croner
Rust `cron` crate timezone 弱（基本只 UTC + local）；`croner` crate 原生支持 IANA 时区 + DST。`chrono`（runtime 已依赖）作为时间类型基础。新增 dep：`croner`。

DST 行为（记下来防止未来读者以为是 bug）：croner 对 spring-forward gap 里的 fixed-time job 是 **snap to end of gap**，不 skip。例如美国 LA 在 2026-03-08 跳过 02:00–03:00 那一小时，cron `30 2 * * *` 那天会 fire 在 03:00 LA（PDT），不是 skip 整天。这是 croner 默认行为，gitim-core 的 `dst_forward_no_double_fire` 测试 assert 的就是这个语义。

### Catch-up 策略：不补跑
runtime/daemon 当时关机错过的 fire = 错过。补跑会让重启后瞬间炸一堆任务（烧 token / 撑爆 context）。

错过的可见性走**前端实时计算**：calendar UI 拉 `timeline` endpoint 时，runtime 比对（spec.schedule × 时间窗）与 `<ts>.thread` 实际文件，差集即 missed。daemon 不写任何 mutable state，git log 干净。

### Fire 失败处理（target 不存在 / fs 错 / lock timeout）
- v1：daemon log 记一条，跳过这次 fire（下个 schedule 周期再试）
- v2：写 `crons/<name>/<ts>.failed` 占位文件 + 错误信息，calendar 可显示

### Busy 时怎么处理：零额外代码
fire = 消息进 thread = poller 看到 ChannelChange = agent_loop 处理。如果 agent 此刻 busy（一个 session 在跑），新事件进 `session.events` queue，下个 cycle 自然消费。已有机制，cron 不需要任何特殊路径。

### Timezone 一致性
spec 用 IANA 字符串。daemon 解析时区基于 IANA 数据库，所有机器结果一致 — 物理事实，不需特殊处理。v1 不引入 workspace 全局默认时区（v2 看需要）。

## CLI 层：AI 自助创建的主路径

AI agent 不应手写 yaml — 它不知道字段规范，容易出错。CLI 是 AI **和** 人类共用的创建路径。

### 命令集（v1）
```
gitim cron create <name> --schedule <expr> --target <handler> \
  (--prompt <text> | --prompt-file <path>) [--timezone <tz>]
gitim cron list                         # active list（不含 archive）
gitim cron show <name>                  # spec + 最近 5 次执行 + 下次时间
gitim cron history <name>               # 历史 .thread 文件列表
gitim cron disable <name>               # 暂停（保留 spec，不 fire）
gitim cron enable <name>                # 恢复
gitim cron delete <name>                # 软删除 → git mv 到 archive/crons/
gitim cron next <name>                  # 算下一次 fire 时间（用户验证用）
```

`--prompt-file` 是多行 prompt 的备选（避免 shell quoting 之苦）。

### 流程
CLI 经 Unix socket → daemon → 新加的 `crates/gitim-daemon/src/handlers/cron.rs`。Daemon 校验：
- `<name>` 唯一性 + 命名合法（小写 a-z 0-9 连字符）
- `--schedule` cron 表达式有效（用 `croner` 解析）
- `--target` handler 在当前 workspace 存在（在 `users/` 里）；`@self` 解析为创建者实际 handler
- `--timezone` IANA 有效
- `--prompt` 非空，长度合理

校验通过 → 写 spec.yaml + commit + 触发 sync 推送。

**v1 无 authorization 模型**：任何 handler 可 create 任何 target 的 cron。builder 阶段可接受，v2 看需求。

### 终止操作的语义对照
| 操作 | spec.yaml | 历史 .thread | active list 显示 | 可恢复 |
|---|---|---|---|---|
| `disable` | 加 `enabled: false` | 保留原位 | 显示但标灰 | ✅ `enable` |
| `delete` | git mv 到 `archive/crons/<name>/` | 跟着移走 | 不显示 | 手动从 archive 移回 |
| `purge`（v2） | `git rm -rf` | 一并删 | 不显示 | ❌ |

不直接物理删的理由：cron 历史是审计资产（"上个月那次失败用了什么 prompt"），物理删意味着 `git rm + commit`，git history 仍能找回但很别扭。软删走 `archive/crons/<name>/` 与 [crates/gitim-daemon/src/handlers/channel.rs:221](crates/gitim-daemon/src/handlers/channel.rs:221) 的 channel-archive 实现保持一致。

## Agent system prompt 注入

agent 的 system prompt 模板加一段（具体位置：`crates/gitim-runtime/src/agent_loop.rs` 或 prompt 模板源），用交接语气而不是程序化清单（参考 `feedback_prompt_style_for_llms.md`）：

> 你可以给自己或别的 agent 安排定期任务。例如：
>
> ```
> gitim cron create weekly-summary \
>   --schedule "0 9 * * 1" \
>   --target @self \
>   --prompt "扫一下上周 #general 的关键讨论，整理成周报发到 #general"
> ```
>
> 时间表达式是标准 5-field cron。`gitim cron list` 看当前都有什么。
> 被 `[@system] cron(<name>) ...` 消息唤醒就是定期任务到点；处理完后回一条 "做完了 + 简要日志" 到那条消息所在 thread，方便回溯。

## WebUI 层：日历视图

WebUI v2 加 `/crons` tab，月历组件展示历史 + 未来预测 fire + missed。点某天 → 当天事件列表。点事件 → 历史 fire 打开 .thread viewer / 未来 fire 打开 spec detail / missed fire 显示原因。

### Runtime HTTP API
```
GET /workspaces/<slug>/crons                       # active job 列表 + schedule + last/next fire
GET /workspaces/<slug>/crons/<name>                # job 详情
GET /workspaces/<slug>/crons/<name>/runs           # 该 job 的 .thread 列表（按时间倒序）
GET /workspaces/<slug>/crons/<name>/runs/<ts>      # 单次执行 thread 内容
GET /workspaces/<slug>/crons/timeline?from=&to=    # 时间窗内：历史 fire + 未来预测 + missed，merge
```

`timeline` 是关键端点，返回三类条目：
- 历史段：glob `crons/*/<ts>.thread`，按时间戳过滤
- 未来段：每个 active spec 解析 cron 表达式，算出 `[from, to]` 内的所有 next fire
- Missed 段：未来段中**早于 now** 的 timestamp 在历史段中找不到对应文件 = missed

merge 排序返回。前端拉一次 `timeline` 画整月日历。所有 clone 看到的日历一致（数据在 git 里）。

### v1 范围
- 只读视图（看 + 浏览历史 + 看 missed）
- "修改 spec / 创建 cron" 在 WebUI 是 v2（v1 走 CLI）

## 失败与边界

### daemon 在 fire 中途被 kill
fire 在 `commit_lock` 内执行：
- kill 在 lock 之前 → 漏 fire（按 catch-up：不补，前端可显示 missed）
- kill 在 lock 之后 → thread 已写进 git index，重启时 sync_loop 自然推送
- **不会出现"消息一半写进去一半没"的中间状态**

### Spec 改动正赶上 fire 中
fire 拿 lock 时已 snapshot spec 内容；后续改动影响下一次 fire，不影响当前。

### 删除一个 active job 时正好赶上它在 fire
当下这次执行不打断（消息已写进 thread，agent 自然处理完）。下一次 fire 不会发生。

### 多 clone 同 workspace（v1 常态）
两台机器各 onboard 一个不同 handler 到同一 workspace 是 v1 正常用法（Lewis 用机器 A 当 @lewis，Bob 用机器 B 当 @bob，同一 workspace）。每台机器有自己的 daemon + cron_engine。

如果 daemon 不加过滤就 fire 所有 spec，会 double-fire。**ownership 过滤就是为这个场景**：每个 daemon 只 fire `target == 自己 me.json 里 handler` 的 spec。spec for `@lewis` 只在机器 A 上 fire；机器 A 关机 = miss（与 catch-up 一致）。

同 handler 跨机器（machine A 和 B 都自称 @lewis）已被 `add_agent` 的 `handler_conflict` 防护，不在此 feature 范围。

### sync_loop 非 thread filter（先决条件）
[crates/gitim-sync/src/sync_loop.rs:332](crates/gitim-sync/src/sync_loop.rs:332) `diff_unpushed("*.thread")` 只追踪 thread 文件作为 local additions。spec.yaml 改动在 fast-path push 时正常，但 rebase 冲突路径下行为不明（参考 `project_sync_loop_non_thread_bug.md`）。

**实施 step 0**：先单独跑一个 e2e 场景 — 两台机器同时改同一 `crons/<name>/spec.yaml`，确认 sync_loop 处理（先到先赢 / 冲突报错都行，不能静默丢一边）。如果丢，先修 sync_loop 再做 cron — 非 thread 文件场景全 workspace 受益。

### renumber 对 cron thread 的处理
[crates/gitim-sync/src/renumber.rs](crates/gitim-sync/src/renumber.rs) 用 `parse_thread` 逐文件处理；理论上 `crons/<name>/<ts>.thread` 各自独立，line numbers 内部独立，不冲突。**实施时加测试**确认两台机器同时写不同 `<ts>` thread 时 renumber 不会误识别。

## 备选方案对照

### Approach A（采纳）：协议一等公民 + 合成消息
如本文。新协议目录 + 复用消息机制 + daemon 内 cron engine + ownership 过滤 + CLI 主路径 + 日历 UI。

### Approach B（拒绝）：暗线 fire
runtime 内部直接喂 prompt 给 agent_loop，跳过 thread。
- ✗ 失去 git 审计能力
- ✗ runtime 关机时纯丢 fire，无可见性
- ✗ 多 runtime 协作复杂（需 leader election）
- ✗ 破坏"agent 行为通过 thread 流动"的核心 invariant

### Approach C（拒绝）：cron 字段挂在 `users/<handler>.meta.yaml`
把 cron 表达式塞进现有 UserMeta。
- ✗ UserMeta 是身份字段，不是行为字段
- ✗ 一个 agent 多个 cron 时 schema 难看
- ✗ 跨 agent 调度破坏 ownership
- ✗ 历史 fire 没地方存

## v1 non-goals

- `gitim cron purge`（物理删）
- workspace 全局默认时区
- in-chat cron 创建（"@scheduler 创建一个..."）— 需要专门 agent
- 失败重试 / `concurrency` 字段（"已经在跑就跳过/排队"）
- 跨 channel cron（同一 fire 触发多 channel）
- WebUI 日历的"修改 spec / 创建 cron"操作（v1 只读）
- catch-up（错过自动补跑）
- file watcher（v2 替每分钟扫描 — 用 `notify` crate）
- Authorization 模型（v1 任何 handler 可 create 任何 target 的 cron）
- "Done" 信号约定（v1 不约定，UI 显示 thread 内容）
- Failure marker 文件（`<ts>.failed`）
- 同分钟批量 commit 优化

## plan-eng-review 已解决的决策

| 议题 | 决策 |
|---|---|
| Multi-clone fire collision | engine 加 ownership 过滤（target == self_handler 才 fire） |
| Idempotency 机制 | `<theoretical_ts>.thread` 存在性 = fire 凭证 |
| `@self` 解析时机 | create-time 替换为创建者具体 handler |
| Archive 路径 | `archive/crons/<name>/`（top-level，与 channel 一致） |
| `last_missed_at` mutable state | 不存，UI 实时算 |
| Cron 库 | `croner`（替 `cron`） |
| Bootstrap fire | 不立刻，`last_fire = created_at` 后等 next match |
| sync_loop 先决条件 | implementation step 0 先验证；丢就先修 sync_loop |
| Authorization | v1 无（any handler can create any cron） |
| Fire failure visibility | v1 daemon log only |
| Done 信号 | v1 不约定 |
| `--prompt-file` CLI 选项 | 加（多行 prompt） |
