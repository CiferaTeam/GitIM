## 问题

gitim agents 当前是完全被动的 — 只在被 @-mention 或收到 DM 时唤醒。需要一种机制让 agent 在固定时间点自主行动，例如"每周一 9 点写周报"、"每天上班前扫一下 GitHub PR"、"每小时刷一次外部 API 信号"。

## 核心 framing：cron = 协议一等公民 + 复用消息机制

Cron 是协议级别的新目录 `crons/`，但触发行为完全复用 thread + 消息 + agent_loop 现有路径。到点时 daemon 假装自己叫 `@system` 把一条 prompt 写进对应 thread 文件，目标 agent 通过现有 poller 路径醒来处理。

这保留了 gitim 的核心 invariant —— **agent 行为通过 thread 消息流动** —— 而不是开外挂破坏它。

## 协议层：文件布局

```
<workspace>/
  crons/
    weekly-report/
      spec.yaml                       # cron 表达式 + target + prompt 模板
      2026-05-11T09-00-00Z.thread     # 周一 9 点这一次执行
      2026-05-18T09-00-00Z.thread     # 下周一这一次执行
    daily-standup/
      spec.yaml
      2026-05-11T09-30-00Z.thread
    .archive/
      old-job/
        spec.yaml
        2026-04-...thread
```

每个 cron job 一个文件夹。`spec.yaml` 是配置；每次执行一个独立 `.thread` 文件，文件名嵌时间戳（ISO8601，`:` → `-` 兼容文件系统）。

### 为什么"每次执行一个 thread"，而不是"一个 job 一个长 thread"
- 每次执行是独立上下文（fire → start → done），不该混在一个长 thread
- 长 thread 永远涨，renumber 越来越贵
- "重跑某次"/"查那个周一发生了什么" — 独立文件干净
- 前端展示成本 0：`ls crons/<job>/` 看 job 历史；glob `crons/*/2026-*.thread` 看时间线，文件名直接是排序键

代价：git 仓库会涨。但 cron 日志本来就是高 churn，可接受。v2 加 prune 策略（保留最近 N 次）。

### spec.yaml 字段（最终以 Rust 类型为准，参考 `feedback_code_is_contract.md`）
- `schedule`: 标准 5-field cron + alias（`@daily/@hourly/@weekly`）
- `timezone`: IANA 字符串（`America/Los_Angeles`），缺省 UTC
- `target`: 目标 handler（`@analyst` / `@self`）
- `prompt`: 触发时写进 thread 的 body
- `enabled`: bool，默认 true（用于 disable）
- `archived`: bool（软删除）+ `archived_at` + `archived_by`
- `created_by`, `created_at`
- `last_missed_at`: 可选，记录最近一次错过的 fire 时间

具体 Rust 类型落 `crates/gitim-core/src/types/cron.rs`，配解析 + 校验 + tests。

### 为什么不用现有 `users/<handler>.meta.yaml`
- UserMeta 是身份字段，不是行为字段
- 一个 agent 多个 cron 时 schema 难看
- 跨 agent 调度（一个 agent 给另一个加 cron）会破坏 ownership 语义
- 历史 fire 记录没有合适的协议位置

## Runtime / Daemon 层：cron 引擎

### 引擎位置：daemon 内部
一个 daemon-managed tokio 任务，每分钟扫所有 `crons/*/spec.yaml`，找出"上次扫到现在"区间内应该 fire 的 spec，挨个 fire。

为什么不在 runtime：
- daemon 是 workspace 单一 writer + 单点 — 多 runtime 不会重复 fire
- daemon 已有 `commit_lock`（参考 `project_commit_tree_lock.md`），cron fire 直接复用
- runtime 不该背 clock 责任 — 它的职责是驱动 agent_loop

### Fire 流程
1. cron engine 算出 spec X 应该 fire（now ≥ next_scheduled_fire）
2. 拿 `commit_lock`
3. 写 `crons/<name>/<timestamp>.thread`，首行：`[L1][@system][<ts>] cron(<name>): <prompt>`
4. 加入 git index
5. 释放 lock
6. sync_loop 自然推送（不需特殊路径）
7. 目标 handler 的 runtime 通过 poller 拿到 ChannelChange，`format_changes_as_prompt` 生成 prompt，agent 醒来处理

### Catch-up 策略：不补跑
runtime/daemon 当时关机错过的 fire = 错过。补跑会让重启后瞬间炸一堆任务（烧 token / 撑爆 context）。

但写 `last_missed_at` 到 spec.yaml（或 `crons/<job>/<ts>.missed` 占位文件）让前端可显示"昨天 9 点错过了"，不静默丢。具体落点（spec 字段还是单独 .missed 文件）见"开放问题"。

### Busy 时怎么处理：零额外代码
fire = 消息进 thread = poller 看到 ChannelChange = agent_loop 处理。如果 agent 此刻 busy（一个 session 在跑），新事件进 `session.events` queue，下个 cycle 自然消费。已有机制，cron 不需要任何特殊路径。

### 多 workspace / 多机器的 timezone 一致性
spec 用 IANA 字符串。daemon 解析时区基于 IANA 数据库，所有机器结果一致 — 物理事实，不需特殊处理。v1 不引入 workspace 全局默认时区（v2 看需要）。

## CLI 层：AI 自助创建的主路径

AI agent 不应手写 yaml — 它不知道字段规范，容易出错。CLI 是 AI **和** 人类共用的创建路径。

### 命令集（v1）
```
gitim cron create <name> --schedule <expr> --target <handler> --prompt <text> [--timezone <tz>]
gitim cron list                         # active list（不含 archive）
gitim cron show <name>                  # spec + 最近 5 次执行 + 下次时间
gitim cron history <name>               # 历史 .thread 文件列表
gitim cron disable <name>               # 暂停（保留 spec，不 fire）
gitim cron enable <name>                # 恢复
gitim cron delete <name>                # 软删除 → .archive/
gitim cron next <name>                  # 算下一次 fire 时间（用户验证用）
```

### 流程
CLI 经 Unix socket → daemon → 新加的 `crates/gitim-daemon/src/handlers/cron.rs`。Daemon 校验：
- `<name>` 唯一性 + 命名合法（小写 a-z 0-9 连字符）
- `--schedule` cron 表达式有效（用 `cron` crate 解析）
- `--target` handler 在当前 workspace 存在
- `--timezone` IANA 有效
- `--prompt` 非空，长度合理

校验通过 → 写 spec.yaml + commit + 触发 sync 推送。

### 终止操作的语义对照
| 操作 | spec.yaml | 历史 .thread | active list 显示 | 可恢复 |
|---|---|---|---|---|
| `disable` | 加 `enabled: false` | 保留原位 | 显示但标灰 | ✅ `enable` |
| `delete` | 移到 `crons/.archive/<name>/` | 跟着移走 | 不显示 | 手动从 archive 移回 |
| `purge`（v2） | `git rm -rf` | 一并删 | 不显示 | ❌ |

不直接物理删的理由：cron 历史是审计资产（"上个月那次失败用了什么 prompt"），物理删意味着 `git rm + commit`，git history 仍能找回但很别扭。软删 + archive 目录复用 `2026-04-03-channel-archive.md` 的 channel-archive 套路，保持一致。

## Agent system prompt 注入

agent 的 system prompt 模板加一段（具体位置：`crates/gitim-runtime/src/agent_loop.rs` 或 prompt 模板源），用交接语气而不是程序化清单（参考 `feedback_prompt_style_for_llms.md`）：

> 你可以给自己或别的 agent 安排定期任务。例如：
>
> ```
> gitim cron create weekly-summary \
>   --schedule "0 9 * * 1" \
>   --target @self \
>   --prompt "扫一下上周 #general 的关键讨论，整理成周报发到 #general"
> ```
>
> 时间表达式是标准 5-field cron。`gitim cron list` 看当前都有什么。
> 被 `[@system] cron(<name>) ...` 消息唤醒就是定期任务到点；处理完后回一条 "做完了 + 简要日志" 到那条消息所在 thread，方便回溯。

## WebUI 层：日历视图

WebUI v2 加 `/crons` tab，月历组件展示历史 + 未来预测 fire。点某天 → 当天事件列表。点事件 → 历史 fire 打开 .thread viewer / 未来 fire 打开 spec detail。

### Runtime HTTP API
```
GET /workspaces/<slug>/crons                       # active job 列表 + schedule + last/next fire
GET /workspaces/<slug>/crons/<name>                # job 详情
GET /workspaces/<slug>/crons/<name>/runs           # 该 job 的 .thread 列表（按时间倒序）
GET /workspaces/<slug>/crons/<name>/runs/<ts>      # 单次执行 thread 内容
GET /workspaces/<slug>/crons/timeline?from=&to=    # 时间窗内：历史 fire + 未来预测，merge
```

`timeline` 是关键端点：
- 历史段：glob `crons/*/[ts].thread`，按时间戳过滤
- 未来段：每个 active spec 解析 cron 表达式，算出 `[from, to]` 内的所有 fire，merge 进结果

前端拉一次 `timeline` 画整月日历。所有 clone 看到的日历一致（数据在 git 里）。

### v1 范围
- 只读视图（看 + 浏览历史）
- "修改 spec / 创建 cron" 在 WebUI 是 v2（v1 走 CLI）

## 失败与边界

### daemon 在 fire 中途被 kill
fire 在 `commit_lock` 内执行：
- kill 在 lock 之前 → 漏 fire，按 catch-up 策略记 `last_missed_at`
- kill 在 lock 之后 → thread 已写进 git index，重启时 sync_loop 自然推送
- **不会出现"消息一半写进去一半没"的中间状态**

### Spec 改动正赶上 fire 中
fire 拿 lock 时已 snapshot spec 内容；后续改动影响下一次 fire，不影响当前。

### 删除一个 active job 时正好赶上它在 fire
当下这次执行不打断（消息已写进 thread，agent 自然处理完）。下一次 fire 不会发生。

### 多 runtime 同 workspace（异常状态）
不该发生（`add_agent` 的 `handler_conflict` 防护）。即使发生，daemon 是单 writer 不会重复 fire；多 runtime 重复处理同一 fire 是 split-brain 老问题，不在 cron feature 范围。

### sync_loop 静默丢非 thread commit 的旧 bug
参考 `project_sync_loop_non_thread_bug.md`。spec.yaml 修改是非 thread commit。如果该 bug 未修复，cron spec 修改可能丢失。**这是先决条件**，需在 implementation phase 确认或修复。

### renumber.rs 的影响
`crons/<job>/<ts>.thread` 文件名固定（嵌时间戳），不应被 renumber。需在 implementation 阶段确认 renumber 路径不会误处理 `crons/` 目录。

## 备选方案对照

### Approach A（采纳）：协议一等公民 + 合成消息
如本文。新协议目录 + 复用消息机制 + daemon 内 cron engine + CLI 主路径 + 日历 UI。

### Approach B（拒绝）：暗线 fire
runtime 内部直接喂 prompt 给 agent_loop，跳过 thread。
- ✗ 失去 git 审计能力
- ✗ runtime 关机时纯丢 fire，无可见性
- ✗ 多 runtime 协作复杂（需 leader election）
- ✗ 破坏"agent 行为通过 thread 流动"的核心 invariant

### Approach C（拒绝）：cron 字段挂在 `users/<handler>.meta.yaml`
把 cron 表达式塞进现有 UserMeta。
- ✗ UserMeta 是身份字段，不是行为字段
- ✗ 一个 agent 多个 cron 时 schema 难看
- ✗ 跨 agent 调度破坏 ownership
- ✗ 历史 fire 没地方存

## 距离 v1 完成的高层分工（具体 task plan 由 writing-plans 阶段产出）

1. **协议层**：`crates/gitim-core/src/types/cron.rs` — `CronSpec` / `CronEntry` / 解析 / 校验 / tests
2. **Daemon 层**：
   - `crates/gitim-daemon/src/handlers/cron.rs` — IPC handlers（create/list/show/disable/enable/delete）
   - daemon 内 cron engine（tokio 任务）— 每分钟扫 spec、算 due、fire（拿 commit_lock）
3. **Sync 层**：确认 sync_loop / renumber 不误处理 `crons/` 目录；如有 sync_loop 静默丢非 thread commit 的旧 bug，先修
4. **CLI 层**：`crates/gitim-cli/src/cron.rs` — 命令解析 + 调 client + unit tests
5. **Client 层**：`crates/gitim-client/src/lib.rs` — 加 cron 相关 API 方法
6. **Runtime 层**：`http.rs` 加 cron 路由（list/show/runs/timeline）+ agent system prompt 注入 cron 用法
7. **Frontend**：日历 tab + timeline 组件（`products/gitim/frontend/`）

每步配 tests，每完成一项就 commit（参考 `feedback_commit_each_task.md`）。

## v1 non-goals

- `gitim cron purge`（物理删）
- workspace 全局默认时区
- in-chat cron 创建（"@scheduler 创建一个..."）— 需要专门 agent
- 失败重试 / `concurrency` 字段（"已经在跑就跳过/排队"）
- 跨 channel cron（同一 fire 触发多 channel）
- WebUI 日历的"修改 spec / 创建 cron"操作（v1 只读）
- catch-up（错过自动补跑）

## 开放问题（留给 plan-eng-review）

1. **`last_missed_at` 落点**：写在 spec.yaml 的字段里（让 spec 有 mutable 状态），还是单独 `crons/<job>/<ts>.missed` 占位文件（更纯但多文件）？倾向后者。
2. **agent 完成信号**：agent 处理完 cron 后回什么样的"完成消息"才能让前端清晰判断"这一次结束了"？需约定一个 done 信号格式，或者不约定让 agent 自由发挥。倾向不约定。
