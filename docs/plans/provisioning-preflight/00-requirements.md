## 背景

[runtime-cli 落地后的已知 gap](../runtime-cli/00-requirements.md) + `docs/specs/runtime-cli.md` "Known gap" 章节明确：

**现状**：
- `POST /workspaces/{slug}/agents/add` 后 server 跑 `provision_agent`（git clone + me.json + .env + hermes profile），然后 `spawn_agent_loop`，**不** 验证"this agent with this config 真的能跑 LLM turn"
- WebUI `add-agent-dialog.tsx` 用户点 "detect" 才让 add —— 但 detect 只调 `/preflight/{provider}` 验**系统级安装环境**（用 default model / default env），**不**用 agent 即将拥有的 specific config
- CLI `gitim-runtime add-agent`（runtime-cli feature 刚 land）没有这一步，更裸

**Gap**：env / model 配错（PAT 失效、model 名拼错、hermes LLM provider 没配等），不会在 add 阶段暴露 —— agent 成功加入 workspace，第一个 turn 在 agent_loop 里才 error。User 视角：add 成功 → agent 静默坏掉 → 看不出在哪一步出问题。

**用户原话**（office-hours Q1）：
> "preflight 只是验了一下本地安装环境的可用性，但并不一定加上这个模型配置之后还是可用的。我真正的 preflight 就是应该在创建的时候去做一次测试。"
> "创建并不是那么高成本的事情，所以多浪费一个对话去做这件事情，其实也没什么问题。"

---

## 关键洞察（office-hours 收敛）

现有 `preflight_claude_with` (preflight.rs:246) 已经是真 LLM round-trip —— spawn `claude --print --model X --system-prompt "..." "Reply with: GITIM_OK"`。**不是** binary 安装版本检查。

那 design scope 缩到 1 行 reframe：
> **现有 preflight_X 加 agent-aware 参数（env + model + hermes_home），在 `provision_agent` 后 `spawn_agent_loop` 前调用一次，失败则 rollback**。

不引入 trait method、不新加 endpoint、不动 agent_loop。

## 共识 Premises

### P1 — 复用现有 preflight 命令，加 agent-aware 变体
不引入新机制。给 `preflight_X_with` 系列加可选参数 `env: Option<&HashMap<String, String>>` + `model: Option<&str>` —— spawn provider CLI 时注入。Hermes 已有 `hermes_home` 参数，传 agent 自己的 profile dir 即可。

具体改动面（per provider）：
- claude: `preflight_claude_with(bin, timeout, env_override, model_override)`
- codex / opencode / pi: 同模板
- hermes: `preflight_hermes_with` 已有 `hermes_home` + `llm_provider` + `llm_model`，已够，**只补 env_override 即可**

### P2 — Sync HTTP，`agents_add` 等 preflight 完才返
P2 的选择基础：user 之前说"创建不是高成本，多浪费一个对话没事"。Sync 对 caller 简单（一个 request → 一个完整结果），async 要加 status poll endpoint + state machine，不划算。

预期延迟：单 turn LLM call ~5-15s（claude）/ ~3-10s（hermes 取决于 LLM）。算上 git clone 已经 30s+ 是常态，preflight 加几秒不显眼。

### P3 — Rollback boundary = undo provision_agent 全副作用
失败时（按顺序）：
1. `cleanup_agent_dir(agent_clone_path)` —— 删 agent clone 目录（rm -rf）
2. 如果 provider == hermes: `delete_profile(profile_name)` —— 删 hermes profile（best-effort，失败仅 warn）
3. **不**加 agent 到 `state.workspaces[slug].agents`
4. **不** spawn agent_loop
5. 返 `error_code: "provision_preflight_failed"`，body 含 `PreflightResult` 的 detail（让 caller 区分 timeout / not_installed / 真 LLM 拒绝）

这跟现有 `hermes_profile_create_failed` rollback 是同一个 pattern（CLAUDE.md "回滚保证" 提过）。复用 `cleanup_agent_dir` + `delete_profile` 两个函数即可。

### P4 — WebUI 删 client-side detect，server preflight 单一权威
**现状**：`products/gitim/frontend/src/components/management/add-agent-dialog.tsx` 让用户先点 "detect" 跑 `/preflight/{provider}`，通过后才让 Submit。

**改后**：删 detect 步骤。"Add agent" 按钮直接 POST `/agents/add`，前端进入 loading 状态，server 跑 preflight 完后返完整结果。失败 → 显示 `error_code` 对应 user-friendly 错误信息（含 `output_preview` 帮助 debug）。

User 体感差异：
- Before: click detect → wait 10s → check passed → click Submit → wait 30s (clone) → done = 2 click, ~40s total
- After: click Submit → wait 40-60s (clone + preflight) → done = 1 click, ~50s total

User 已 commit 这条："前端那块问题不大，到时候简单一改就行" —— v1 实施同步改 WebUI。

### P5 — `/preflight/{provider}` HTTP endpoint 不动
该 endpoint 仍有用 case：
- agent operator 在 add 前手动 sanity check 环境（用 CLI `gitim-runtime preflight <provider>`）
- 未来 WebUI "诊断" 页面可能用
- 不破坏向后兼容

新 preflight 路径走 `agents_add` 内部，**不**暴露独立 endpoint。

### P6 — 协议层零改动
- `users/<handler>.meta.yaml` schema 不动
- `.gitim/me.json` 不动
- `.thread` 不动
- **不**加 git-tracked "preflight result" 字段（preflight 结果是 ephemeral，不需要 audit trail）

### P7 — Provider scope = v1 已支持全部
- claude / codex / opencode / pi / hermes 全部纳入
- mock provider 走 always-pass shortcut（测试时用）
- 未来 provider（如 gemini 直连、ollama 等）加入时需同步实现 agent-aware preflight

### P8 — 现有 `/preflight/{provider}` 行为不变
保持向后兼容：generic preflight（不带 agent env / model）仍然走老 path。Agent-aware preflight 是新的 internal API，不影响 HTTP 表面。

## Considered Alternatives

### Approach A — 复用 preflight_X_with + agent-aware 参数（已选）
**Why chosen**: user 明确说"和现在的 preflight 一样"。最小改动面，preflight.rs 增量 ~50 行，http.rs 增量 ~30 行。

### Approach B — Provider trait 加 `hello(&ProviderConfig) -> Result` method
❌ 否决理由：
- 引入新抽象：trait method 是 cross-cutting，每个 provider impl 都要加
- 行为分裂：每个 provider 实现 hello 的语义可能漂移
- agent_loop 已有"start a turn"路径，再加一个 hello path 是冗余

### Approach C — Hybrid trait method + default impl = agent_loop synthetic turn
❌ 否决理由：
- agent_loop 携带太多 state（poller state、commit_lock、sync hook、cache）
- "仅 preflight 调一次"的 cut-down 版本要拼出来不容易
- Control flow 复杂

### Approach D — Async preflight + `/agents/{id}/preflight_status` poll endpoint
❌ 否决理由：
- HTTP API 复杂化：state machine（pending / running / passed / failed）
- WebUI 也要改 polling 逻辑
- user 说"sync 完全可以接受"
- 增加 server-side state（preflight 进行中的 agent_id 集合）

## Implementation surface

### `crates/gitim-runtime/src/preflight.rs`
- 给 `preflight_claude_with` / `preflight_codex_with` / `preflight_opencode_with` / `preflight_pi_with` 各加 2 个可选参数：
  - `env_override: Option<&HashMap<String, String>>` —— 合并进 `tokio::process::Command.envs(...)`
  - `model_override: Option<&str>` —— 替代 `CLAUDE_PREFLIGHT_MODEL` 等 hardcoded default model
- `preflight_hermes_with` 已有 `hermes_home` + `llm_provider` + `llm_model`，**只补 env_override** 让 hermes profile 启动时也能拿到 agent 的 env vars
- 新增 helper `preflight_for_agent(provider: &str, agent_dir: &Path, config: &ProviderConfig) -> PreflightResult`：
  - 读 agent 的 `.env` 文件（per CLAUDE.md "chmod 0600、64KB 上限"）解析成 HashMap
  - 合并 me.json 的 env field（me.json 优先于 .env）
  - 根据 provider 分发到对应 `preflight_X_with(..., env_override, model_override)`
  - 失败任一步：返 `PreflightResult::failure`

### `crates/gitim-runtime/src/http.rs::agents_add`
- 在 `provision_agent` 成功后、`state.workspaces[slug].agents.insert(...)` 前，新增 preflight 步骤：
  ```
  let preflight = preflight_for_agent(provider, agent_dir, &provider_config).await;
  if !preflight.available {
      cleanup_agent_dir(&agent_dir);
      if provider == "hermes" { delete_profile(&profile_name).await; }
      return Json(ErrorBody::with_code(
          format!("provision_preflight_failed: {}", preflight.error.unwrap_or_default()),
          "provision_preflight_failed",
      )).into_response();
  }
  // continue to spawn agent_loop
  ```

### `products/gitim/frontend/src/components/management/add-agent-dialog.tsx`
- 删除 `detectResult` / `generation counter` / "Detect" 按钮 / `preflightProvider` 调用 (lines 18, 48-49, 153, 177 等)
- "Add" 按钮直接调 `client.addAgent(...)`
- Loading state 显示 "Provisioning agent (this includes a real LLM hello check)..." 或类似
- Error display: 当 response 含 `error_code: "provision_preflight_failed"` 时，展示 PreflightResult 的 `output_preview` + `error` + `error_kind` 帮 user debug

### 测试
- `crates/gitim-runtime/tests/provision_preflight.rs`（新）：
  - claude provider + valid env + valid model → success
  - claude provider + bogus model name → failure + error_code = `provision_preflight_failed`
  - claude provider + invalid api key in env → failure
  - hermes provider + missing hermes profile → 已有 `hermes_not_setup` 分支会先 fire
  - mock provider → always-pass shortcut → success
  - 失败时验 cleanup：`agent_dir` 不存在 + agent 不在 state.workspaces[slug].agents
- 复用 `tests/common/mod.rs` daemon 隔离基础设施

## Open Questions（留给 plan-eng-review）

1. **agent_dir 路径推断**：preflight 需要读 agent 的 `.env`。agent clone 路径在 `provision_agent` 完成后从哪个 state 字段拿？
2. **env_override 合并优先级**：me.json 的 env field vs .env 文件 vs system env，谁覆盖谁？
3. **PreflightResult 序列化进 error body 的形状**：`ErrorBody` 现状是 `{ok, error, error_code}`。要不要扩展加 `preflight_detail` 字段携带完整 PreflightResult，让 WebUI 能展示 `output_preview`？
4. **Timeout 设置**：preflight 单次 timeout 多少？claude 默认 60s（preflight.rs 现有），新 agent-aware path 用同值还是不同？
5. **mock provider 怎么 always-pass**：跑真 mock provider hello 还是 short-circuit？
6. **Failure 时是否需要 git push 已 commit 的 onboard commits**：`provision_agent` 已 commit 进 agent 自己的 clone（meta.yaml 等）。clean 时只删 agent_dir 行不行，还是要 push 撤销 commit？
7. **Hermes profile 失败 vs preflight 失败的 error_code 区分**：现有 `hermes_profile_create_failed` 是 profile 克隆失败；preflight 失败是 LLM round-trip 失败。两个分开还是合并？
8. **CLI add-agent 显示**：CLI subcommand 收到 `provision_preflight_failed` 时怎么呈现给 agent caller？需要在 docs/specs/runtime-cli.md 加 error_code 表

## 不在本 plan 范围（明确 non-goals）

- **修改 `/preflight/{provider}` HTTP endpoint** —— P5 / P8 说了
- **修改 agent_loop** —— preflight 是 add-time 一次性的，不进 agent_loop
- **Trait-level provider abstraction 改动** —— Approach B/C 否决
- **Preflight 结果持久化** —— P6，不进 git 不进 state
- **Preflight 缓存 / 复用** —— 每次 add 跑一次，简单可靠
- **Async preflight + poll endpoint** —— Approach D 否决
- **保留 client-side detect 双层** —— P4 直接删
- **Preflight 失败时部分 retry** —— v1 全 fail，user 改 config 重新 add
- **加新的 provider** —— 不在 scope

## 成功标准（v1 ship 时要能做到）

1. **真实 LLM 配置错误立刻暴露**：
   - 创建 claude agent 时 PAT 拼错 → add 失败 + error_code = `provision_preflight_failed`
   - 创建 hermes agent 时 LLM provider 配错 → 同
   - agent 不会被加进 workspace + 频道，agent_dir 物理清理
2. **Preflight 通过后 agent 必然能跑首 turn**（barring 极端 race / network blip）
3. **WebUI 体感不退化**：用户依然能在一次 click 后看到清晰成功 / 失败信息（含 LLM 输出 preview）
4. **CLI add-agent 收到 `provision_preflight_failed` → exit 2**（permanent 错，符合 runtime-cli §4 exit code 合约）
5. **零回归**：现有 `/preflight/{provider}` 行为不变；现有 `hermes_profile_create_failed` 等 error_code 不变；CI / cross-compile pipeline 不变
6. **测试覆盖**：每 provider 至少 happy + 1 fail（model 错 / env 错）path
