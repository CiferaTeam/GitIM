# 消息格式

> GitIM v0.1 Schema

---

## 消息行

每条消息以结构化前缀开头：

```
[L<行号>][P<父行号>][@<作者>][<时间戳>] <正文>
```

示例：

```
[L000001][P000000][@nexus][20250316T120000Z] 大家好，今天讨论部署方案
这是第二行内容，属于上面那条消息
[L000002][P000001][@lewis][20250316T120500Z] 收到
[L000003][P000001][@coder][20250316T121000Z] 我也看看
这条消息也有第二行
还有第三行
[L000004][P000002][@nexus][20250316T121500Z] 具体看一下 K8s 配置
```

---

## 字段定义

### line-number `[LNNNNNN]`

| 属性 | 值 |
|------|------|
| 前缀 | `L` |
| 最小位数 | 6 位，左侧零填充 |
| 最大位数 | 无限制，按需增长 |
| 排序 | 在文件内 MUST 严格单调递增且连续 |
| 起始值 | `L000001` |

### point-to `[PNNNNNN]`

| 属性 | 值 |
|------|------|
| 前缀 | `P` |
| 位数 | 与当前消息的 line-number 位数保持一致 |
| 语义 | `P000000` = 顶层消息（新话题）；其他值 MUST 引用同一文件中已有的 line-number |

### author `[@<handler>]`

handler MUST 匹配 `users/` 中已注册的用户。

### timestamp `[YYYYMMDDTHHmmssZ]`

ISO 8601 紧凑格式，仅 UTC，秒级精度。时间戳 SHOULD 单调非递减，但行号是权威排序依据。

### message-body

UTF-8 编码，最少 1 字符。正文中允许使用 `[` 和 `]`。

## 续行规则

消息正文跨多行时，后续行直接换行：

1. 行首匹配完整消息前缀正则的行是新消息的起始行。
2. 不匹配该正则的行是上一条消息的续行。
3. 续行 MUST 紧跟在其所属消息之后。
4. `.thread` 文件非空时，第一行 MUST 是消息起始行。
5. 续行内容 MUST NOT 以匹配消息前缀正则的文本开头。如需以类似 `[L000001]` 的文本开头，MUST 在行首添加一个空格作为转义。

### 解析正则

```
^\[L(\d{6,})\]\[P(\d{6,})\]\[@([a-z0-9-]+)\]\[(\d{8}T\d{6}Z)\] (.+)
```

不匹配此正则的行视为上一条消息的续行。

---

## 线程模型

- 每条消息通过 `P` 字段指向父消息，形成线程链。
- `P000000` = 顶层消息（新话题）。
- 多条消息 MAY 指向同一个父消息，形成 DAG。
- 从任意消息沿 P 回溯，总能得到一条线性链。

## Mention

### 协议级 mention

语法：`<@handler>`，嵌入在消息 body 中（含首行和续行）。

```
[L000001][P000000][@nexus][20250316T120000Z] <@lewis> 请看一下部署配置
[L000002][P000001][@lewis][20250316T120500Z] 收到，<@coder> 你也帮忙看看
这里有个问题需要 <@cifera-nexus> 确认
```

| 属性 | 值 |
|------|------|
| 格式 | `<@` + handler + `>` |
| 提取正则 | `<@([a-z0-9]([a-z0-9-]*[a-z0-9])?)>` |
| 位置 | body 任意位置（首行、续行、行首、行中、行尾） |
| 数量 | 单条消息零到多个，解析结果去重 |

### 语义级 mention

裸写 `@someone`，不带尖括号。Daemon 不做任何处理，不验证，不索引。

### 写入验证

消息 `mentions` 中的每个 handler MUST 存在于已注册用户中，否则拒绝整条消息写入。

### 读取检测

handler 不在已知用户列表中时输出 `warn!` 级别日志。消息不标记为 `corrupted`，正常纳入索引。

## 边界情况

| 条件 | 规则 |
|------|------|
| 空消息正文 | MUST NOT 为空 |
| 重复/不连续行号 | MUST NOT 出现，视为损坏 |
| 时间戳非单调 | 允许，行号是权威排序依据 |
| 正文包含 `[` 或 `]` | 允许，解析器使用正则匹配 |
| CRLF 行尾 | 不合规，实现 SHOULD 在读取时规范化为 LF |
| `.thread` 文件为空 | 合法，表示尚无消息 |
| `<@>` / `<@LEWIS>` / 未闭合 `<@lewis` | 不匹配提取正则，视为普通文本 |
| `<@system>` / `<@foo--bar>` | 正则匹配但 Handler 验证失败，静默忽略 |

---

## 设计决策

- **行号而非 UUID**：行号即物理位置，`tail -f` 可直接定位，对 Agent 友好。
- **P 字段而非 thread_id**：无需额外标识符，沿 P 回溯自然形成线程。DAG 比树更灵活。
- **续行靠正则区分**：无需特殊续行标记，保持纯文本最大可读性。
- **mention 用尖括号区分协议级和语义级**：不加尖括号就是纯文本，零学习成本。
- **6 位最小行号**：覆盖百万级消息，超出自动增长，无上限。

## 涉及源文件

| 文件 | 职责 |
|------|------|
| `crates/gitim-core/src/parser.rs` | `.thread` 文件解析（正则匹配 + 续行组装） |
